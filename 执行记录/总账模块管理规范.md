# 总账模块管理规范

更新日期：2026-04-08

## 1. 文档定位

本文档是“财务管理 > 总账”模块的管理规范与实施基线，不是原型说明，不是接口设计稿，不是数据库表结构设计稿，也不是页面操作手册。

后续凡涉及以下主题的设计、评审、开发、联调、测试与验收，均必须先对照本规范：

- 总账模块
- 新建凭证
- 查询凭证
- 凭证详情
- 凭证修改
- 凭证导出
- 凭证元数据
- 凭证辅助核算联动

当前本文档只规范“凭证链路”，不展开以下能力的真实功能细则：

- 审核凭证
- 总账余额表
- 明细账
- 总分类账
- 项目明细账
- 供应商明细账
- 客户明细账
- 个人明细账
- 数量金额明细账

上述能力虽属于总账模块范围，但后续若要真实化，必须先补专题规范或修订本规范后再实施。

本文档自发布之日起，作为总账模块后续工作的唯一实施基线。

## 2. 模块定义

总账模块是财务域内部的核心账务处理中心，负责承载凭证的录入、查询、查看、修改以及后续账簿与余额类能力的基础数据来源。

总账模块必须服务于以下财务场景：

- 日常凭证录入
- 凭证查询与回溯
- 凭证详情查看
- 未记账未审核凭证的受控修改
- 凭证导出
- 账簿和余额类能力的后续取数基础

总账模块当前不负责以下内容：

- 报销业务审批流
- 财务账套生命周期管理
- 会计档案主数据维护
- 财务报表口径定义
- 下载中心异步导出体系的强制接入

## 3. 核心口径与边界

### 3.1 财务上下文口径

本项目的总账模块统一按“财务全局公司上下文”处理。

即：

- 所有总账页面默认跟随当前财务公司上下文
- 新建、查询、详情、修改、导出都必须按当前公司隔离
- 不允许在总账页内再造一套与财务上下文冲突的独立公司状态
- 不允许跨公司读取、修改、导出其他公司凭证

### 3.2 页面形态口径

总账模块的凭证链路默认沿用当前已落地的财务域页面模式。

即：

- 新建凭证、凭证详情、修改凭证统一复用共享凭证编辑器
- 查询凭证固定为“上方筛选区 + 下方分页列表”
- 详情页通过隐藏明细路由进入，不作为左侧单独菜单项
- 不允许为新建、详情、修改拆出三套不同产品定义

### 3.3 后端域模型口径

总账凭证后端统一走现有凭证域入口：

- `/auth/finance/vouchers`

后续新增能力必须优先在该域下演进，不得另造平行总账凭证接口体系。

### 3.4 当前现状口径

当前总账架构下，凭证链路按以下现状理解并约束后续实现：

- `FinanceNewVoucherView` 已承载“新建 / 详情 / 修改”共享页面
- `FinanceQueryVoucherView` 已承载“筛选区 + 分页列表”查询页
- 凭证状态已锁定为 `UNPOSTED / REVIEWED / POSTED`
- 修改规则已锁定为“仅未记账且未审核凭证可修改”
- 会计科目选项已切换为按当前公司账套科目读取，不再长期依赖硬编码种子作为正式数据源

## 4. 核心对象定义

### 4.1 凭证头

凭证头是总账凭证的主身份对象，用于承载凭证级公共信息。

凭证头必须至少具备以下语义：

- 归属公司
- 会计期间
- 凭证类型
- 凭证号
- 制单日期
- 附件张数
- 制单人
- 备注
- 当前状态

### 4.2 凭证明细

凭证明细是凭证的分录对象，用于承载单条借贷业务内容。

凭证明细必须至少具备以下语义：

- 摘要
- 会计科目
- 借方金额
- 贷方金额
- 辅助核算项
- 行号或展示顺序

### 4.3 凭证状态

凭证状态必须是标准化、可追踪状态，不得使用自由文本状态代替。

当前状态语义统一按以下口径处理：

- `UNPOSTED`：未记账未审核
- `REVIEWED`：已审核未记账
- `POSTED`：已记账

后续若要扩展草稿、作废、冲销等状态，必须先修订规范。

### 4.4 凭证查询结果

凭证查询结果是按“凭证头”维度聚合的结果集，不按分录逐行平铺。

查询结果必须至少具备以下语义：

- 显示用凭证号
- 内部路由主键凭证号
- 凭证类型
- 制单日期
- 会计期间
- 摘要展示值
- 制单人
- 附件张数
- 借方合计
- 贷方合计
- 状态

### 4.5 凭证元数据

凭证元数据是共享凭证编辑器的基础选择项集合。

元数据必须至少覆盖：

- 公司
- 部门
- 员工
- 客户
- 供应商
- 项目分类
- 项目
- 会计科目
- 凭证类型
- 币种

## 5. 总账模块统一业务规则

### 5.1 查询与编辑统一规则

- 总账页面必须先完成财务上下文初始化，再执行查询或编辑
- 所有凭证读写动作都必须校验当前公司与目标凭证所属公司一致
- 凭证详情默认只读，不允许详情页直接进入可写态
- 修改必须是受控动作，不允许以“复制成新凭证”替代“修改原凭证”作为默认实现

### 5.2 共享编辑器规则

- 新建、详情、修改必须共享同一套凭证编辑器
- 共享编辑器必须以显式模式区分 `create / detail / edit`
- 编辑器模式不得依赖隐式路由副作用自行猜测
- 详情模式与修改模式的只读 / 可编辑边界必须明确且稳定

### 5.3 公司切换规则

- 总账页属于财务上下文敏感页面
- 切换财务公司时，当前可见总账页必须先完成确认或丢弃策略，再加载新公司数据
- 隐藏页、缓存页不得在非激活状态下继续发起凭证加载或路由重定向

### 5.4 导出规则

- 凭证导出按当前筛选条件直接导出
- 首版默认不强制走下载中心异步任务
- 导出结果必须与当前筛选条件严格一致
- 不允许前端用当前页数据拼装伪导出结果替代后端正式导出

## 6. 凭证链路重点规范

### 6.1 产品定位

凭证是总账模块中的首要核心对象。

后续以下能力都必须以凭证链路为前置基础：

- 审核凭证
- 总账余额表
- 明细账
- 总分类账
- 各类辅助明细账

凭证不得被视为普通列表记录，也不得拆成与总账脱节的独立业务对象。

### 6.2 新建凭证规则

新建凭证必须满足以下要求：

- 必须基于当前公司账套
- 必须基于当前公司会计科目
- 辅助核算项必须统一按当前公司上下文取值
- 借贷金额必须按标准金额口径录入
- 备注作为凭证级补充说明字段

页面布局统一沿用当前已锁定口径：

- 不保留“附加说明”
- “备注”顶替原附加说明位置
- 凭证编号展示为“凭证类型选择框 - 凭证号输入框”
- 不保留“校验摘要”
- 辅助核算区域全宽且更紧凑
- 分录行号与摘要同行显示
- 分录行高压缩，但不得牺牲可读性

### 6.3 查询凭证规则

查询凭证必须满足以下要求：

- 页面结构固定为“筛选区 + 分页列表”
- 默认不显示统计卡片
- 筛选条件至少包括：
  - 凭证号
  - 凭证类型
  - 制单月份
  - 制单月份起
  - 制单月份止
  - 摘要
  - 下载
- 若填写“制单月份”，则优先生效并覆盖月份区间
- 未填写单月时，再按起止月份区间筛选

列表展示必须按凭证头聚合，不得按分录逐行展示。

摘要口径统一按以下规则处理：

- 过滤时匹配任一分录摘要
- 展示时取第一条非空摘要
- 若存在多条不同摘要，则显示“首条摘要 + 等 N 条分录”

交互口径统一如下：

- 双击整行任意区域进入凭证详情
- 单击仅作选中态，不触发跳转

### 6.4 详情与修改规则

凭证详情必须满足以下要求：

- 默认只读
- 详情页通过独立明细路由进入
- 标题根据模式动态切换为“凭证详情”或“修改凭证”

凭证修改必须满足以下要求：

- 仅 `UNPOSTED` 且未审核凭证允许修改
- `REVIEWED` 与 `POSTED` 凭证默认只读
- 修改按钮仅在有权限且状态可编辑时可见或可用
- 进入修改模式后，保存行为必须更新当前凭证，而不是创建新凭证

修改范围统一锁定为：

- 允许修改：制单日期、附件张数、备注、分录摘要、会计科目、借贷金额、辅助核算等业务内容
- 不允许修改：公司、期间、凭证类型、凭证号等身份维度

更新实现口径统一如下：

- 后端按既有凭证主键定位整张凭证
- 在同一事务内删除原分录并重建新分录
- 保持原 `voucherNo` 不变

### 6.5 状态与流转规则

状态推导统一以现有账务字段语义为准：

- `ibook = 1` 视为已记账
- `ccheck` 非空视为已审核未记账
- 其余状态视为未记账未审核

状态展示文案统一如下：

- `UNPOSTED`：未记账
- `REVIEWED`：已审核
- `POSTED`：已记账

非 `UNPOSTED` 凭证默认只读，不得通过前端绕过状态限制强制进入修改。

### 6.6 元数据与辅助核算联动规则

- 凭证元数据必须与当前财务公司一致
- 会计科目必须来源于当前公司账套科目
- 部门、员工、客户、供应商、项目等辅助核算项必须与当前公司档案口径一致
- 没有账套或账套未激活的公司，不得再回退硬编码科目种子作为正式口径

后续若辅助核算启用规则、必填规则、科目约束规则要深化，必须先补充规范。

## 7. 权限与审计规范

### 7.1 权限语义

总账凭证链路后续至少应规范以下权限能力：

- 查看新建页
- 创建凭证
- 查看查询页
- 查看凭证详情
- 导出凭证
- 修改既有凭证

本次只定义权限语义边界，不在文档中提前写死完整权限树设计稿。

### 7.2 审计要求

以下动作默认视为高风险动作，必须留痕：

- 新建凭证
- 修改凭证
- 导出凭证

审计信息至少应记录：

- 操作者
- 操作时间
- 所属公司
- 目标凭证
- 输入参数摘要
- 操作结果
- 失败原因

对于凭证修改，还必须能回答：

- 修改前的关键字段是什么
- 修改后的关键字段是什么
- 是否改变了分录内容或辅助核算内容

## 8. 与现有系统关系

### 8.1 与会计档案的关系

- 会计档案负责客户、供应商、员工、部门、会计科目等主数据维护
- 总账模块负责消费这些主数据进行凭证录入、查询和修改
- 总账不得自行定义一套与会计档案脱节的辅助核算口径

### 8.2 与财务系统管理的关系

- 财务系统管理负责账套与初始化模板
- 总账模块负责消费账套生成后的公司级科目和账务环境
- 总账不得承担账套创建、模板初始化、账套删除职责

### 8.3 与财务报表的关系

- 财务报表依赖总账数据，但不属于本规范主体
- 本规范只定义凭证链路，报表取数与报表口径后续需另行规范

## 9. 后续实施约束

后续任何总账凭证相关页面、接口、权限、SQL、测试方案，都必须先对照本规范。

未在本规范中明确的高风险行为，默认不得直接实现。

统一实施优先级口径如下：

- 先出规范
- 再按规范拆页面、接口、数据结构、权限和测试
- 不允许先写功能后补规范

后续若要继续落地总账主体能力，建议按以下优先级推进：

- 第一优先级：审核凭证
- 第二优先级：总账余额表
- 第三优先级：明细账与总分类账
- 第四优先级：各类辅助明细账

### 9.1 前端显示与编码稳定性规范

后续凡涉及总账前端页面、权限种子文案、接口返回文案、提示文案、导出结果文案、凭证状态文案、辅助核算显示文案的新增或修改，必须统一按以下规则处理：

- 默认统一使用 UTF-8 无 BOM，不得提交乱码、BOM 污染或混杂编码文件
- 中文文案必须以源码可读中文或统一转码方式稳定保存，不得提交“页面勉强可显示但源码已乱码”的中间态
- 菜单名、页面标题、页签标题、筛选项、表头、按钮、弹窗或抽屉标题、状态文案、错误提示、空态文案必须与总账产品语义一致，不得残留英文占位、乱码占位或临时测试文案
- 页面展示不得依赖硬编码脏数据兜底；若后端字段缺失、异常或辅助核算数据暂不可用，页面必须局部降级并给出可读中文提示，不得直接展示乱码、对象串化结果或技术字段
- 涉及新建凭证、查询凭证、凭证详情、修改凭证、凭证导出、辅助核算选择器、导出文件名、下载记录时，必须验证中文显示一致性

后续提交前，至少必须完成以下前端稳定性自检：

- 菜单可见性
- 页面标题与页签标题
- 筛选区与表头文案
- 弹窗或抽屉标题与表单项
- 状态文案与错误提示
- 辅助核算选项显示
- 导出结果展示与文件名回显
- 页面切换前后文案一致性

未完成上述编码与显示自检的变更，默认不得进入联调或验收。

### 9.2 后端实现与运行稳定性规范

后续凡涉及总账后端接口、服务、元数据、导出、凭证保存与修改链路的新增或修改，必须统一按以下规则处理：

- 必须遵循“DTO 格式校验 + Service 业务校验 + 持久化执行校验”三层校验，不得只依赖单层校验
- 涉及凭证头、凭证明细、会计科目、辅助核算项、凭证状态、导出条件的读取与写入时，必须先校验对象存在性、公司归属、状态合法性、身份字段约束和重复提交风险，不得默认假设数据一定完整
- 新增或修改接口时，必须保证前后端契约稳定；字段名、枚举值、状态语义、时间格式、错误文案、辅助核算返回口径不得随意漂移，若确需调整，必须先修订规范再实施
- 新建、修改、导出等重操作必须保持可追踪、可定位失败原因，不得在 Controller 中直接堆叠高风险长链路逻辑
- 后端异常处理不得把堆栈、技术异常对象、SQL 异常或无语义错误直接暴露给前端，必须转换为可读、可追踪、可定位的中文业务错误
- 新增或修改后端逻辑时，必须同步检查财务上下文、权限判断、公司隔离、凭证状态流转、辅助核算引用关系、账套与科目来源、导出边界和 keep-alive 场景下的运行态影响，避免“当前页可用但系统运行态串扰”
- 对凭证修改、辅助核算变更、导出筛选等受控动作，必须具备幂等和失败收口意识，避免旧请求回写、重复提交或异常中断导致凭证状态错乱或脏数据残留

后续测试与验收至少必须满足以下要求：

- 触及总账前端展示的变更，必须完成文案与编码回归检查，重点覆盖菜单、页面标题、筛选区、列表列名、表单项、状态文案、辅助核算选项、错误提示、导出结果展示
- 触及总账后端逻辑的变更，必须至少补齐对应的 controller 与 service 回归测试，覆盖合法输入、非法输入、重复提交、异常分支、权限分支、公司隔离分支和状态限制分支
- 对新建凭证、修改凭证、导出凭证、元数据联动等关键链路，必须验证成功路径正常、失败路径可追踪、前端收到的错误信息可读且不误导

未完成后端回归测试或异常收口验证的变更，默认不得进入联调或验收。

## 10. 后续任务引用方式

后续每一个总账凭证相关需求，都必须先引用本规范的对应条目再实施。

引用要求如下：

- 需求评审时，必须注明所对应的规范章节
- 页面方案、接口方案、权限方案、SQL 方案、测试方案必须逐项对照本规范
- 凡涉及页面文案、接口字段、凭证状态文案、辅助核算选项、导出结果说明、错误提示、通知文案、任务结果文案，必须同时引用 9.1 和 9.2
- 若需求与本规范冲突，必须先修订规范，再进入实现
- 未引用本规范的总账相关实现，默认视为规格不完整

本文档自发布之日起，作为“总账模块”后续工作的唯一基线文档。


## 数据库字段标准补充

- 涉及数据库建表、加列、改列时，所有表注释和字段注释必须使用真实中文，禁止 `column comment`、`table comment`、纯英文占位和 `?` 乱码。
- 金额字段统一使用 `DECIMAL(18,2)` 作为货币类型标准；费率、汇率、数量、工作量等非金额字段不得误套该规则。
- 新库只执行 `init*.sql`，旧库升级只执行 `migrate*.sql`，禁止手工点改数据库后不留脚本。
