# 报销管理模块管理规范

更新日期：2026-04-08

## 1. 文档定位

本文档是“报销管理”模块的管理规范与实施基线，不是原型说明，不是接口设计稿，不是数据库表结构设计稿，也不是页面操作手册。

后续凡涉及以下主题的设计、评审、开发、联调、测试与验收，均必须先对照本规范：

- 新建报销
- 我的报销
- 待我审批
- 单据查询
- 单据详情共享链路
- 报销管理相关权限、SQL、测试与发布

当前本文档只规范报销管理运行态主链路，不展开以下能力的真实功能细则：

- 流程管理配置中心
- 支付模块与付款单
- 凭证生成与总账链路
- 首页工作台与运营看板
- 下载中心与消息通知的专题产品定义

上述能力虽与报销管理存在上下游关系，但不属于本文档的主体实现范围；后续若要重定义，必须先补专题规范或修订本规范后再实施。

本文档自发布之日起，作为报销管理模块后续工作的唯一实施基线。

## 2. 模块定义

报销管理模块是报销系统的运行态业务主链路，用于承载员工提单、本人单据查看、审批待办处理以及跨单据查询。

报销管理模块必须服务于以下业务场景：

- 选择模板并发起新建报销
- 保存草稿、提交、重提、审批修改
- 查看本人报销单据及其状态
- 查看待我审批任务并执行审批动作
- 按权限范围进行跨单据查询
- 打开统一详情页查看单据头、费用明细、轨迹与评论

报销管理模块当前不负责以下内容：

- 流程模板、表单设计、费用明细设计等配置能力
- 付款执行、银企直连、人工支付工作台
- 财务凭证生成、总账、会计档案
- 与报销主链路无关的第二套报销产品入口

## 3. 核心口径与边界

### 3.1 模块归属口径

报销管理属于现有报销系统工作台内的运行态业务中心，不允许脱离现有菜单体系和平行再造第二套报销产品。

即：

- 菜单入口继续归属现有报销管理侧边栏
- 页面风格继续沿用报销管理工作台的布局、卡片、列表、详情和编辑器体系
- 不允许把“新建报销 / 我的报销 / 待我审批 / 单据查询”拆成彼此风格和语义不一致的独立产品

### 3.2 前端路由口径

当前报销管理前端基线以以下路由链路为准：

- `/expense/create`
- `/expense/list`
- `/expense/approval`
- `/expense/documents`
- `/expense/documents/:documentCode`

同时存在以下共用编辑或明细链路：

- `/expense/documents/:documentCode/resubmit`
- `/expense/approval/tasks/:taskId/modify`
- `/expense/create/details/:detailNo`
- `/expense/documents/:documentCode/details/:detailNo`

后续新增能力必须优先在这条既有路由域下演进，不得为相同业务语义再造第二套路由。

### 3.3 后端接口域口径

报销管理后端统一按当前三组接口域处理：

- 新建报销：`/auth/expenses/create/**`
- 单据与详情：`/auth/expenses/**`
- 审批待办：`/auth/expense-approval/**`

单据查询继续通过：

- `/auth/expenses/query-documents`

后续新增报销运行态能力必须优先在上述接口域下扩展，不得随意另造平行控制器域。

### 3.4 权限口径

报销管理权限统一继续沿用以下命名空间：

- `expense:create:*`
- `expense:list:*`
- `expense:approval:*`
- `expense:documents:*`

后续不得把同一类报销运行态能力散落到无关权限域。

### 3.5 当前现状基线

当前报销管理模块的真实架构基线按以下实现理解：

- `ExpenseCreateView.vue` 承载“新建 / 重提 / 审批修改”共享编辑器
- `ExpenseListView.vue` 承载“我的报销”
- `ExpenseApprovalView.vue` 承载“待我审批”
- `ExpenseDocumentsView.vue` 承载“单据查询”
- `ExpenseDocumentDetailView.vue` 承载统一单据详情页
- 三个列表页继续共享 `expenseWorkbenchListHelper.ts` 的列定义、状态口径与筛选逻辑
- 运行态核心后端服务继续以 `ExpenseDocumentService` 为中心

后续所有扩展都必须建立在这套基线上，不允许忽略现状直接重写一套新体系。

## 4. 核心对象定义

### 4.1 模板

模板是新建报销的入口对象，用于决定单据类型、主表单、绑定流程和费用明细设计等配置来源。

模板在报销管理中是被消费对象，不是由报销管理维护的配置对象。

### 4.2 单据

单据是报销管理的核心运行态对象，用于承载一次报销、申请、借款或合同流程实例。

单据至少应具备以下语义：

- 单据编码
- 单据标题
- 模板名称
- 提单人
- 当前状态
- 当前节点
- 金额
- 提交时间

### 4.3 费用明细

费用明细是报销单的子表单运行态实例，用于承载实际支付、企业往来、票据或费用分摊等子项数据。

费用明细只在模板绑定了费用明细设计时启用，不允许脱离模板配置自行猜测结构。

### 4.4 审批任务

审批任务是待我审批页展示和操作的运行态待办对象，用于承载当前审批人的处理上下文。

审批任务至少应具备以下语义：

- 任务 ID
- 对应单据编码
- 提单人
- 当前审批节点
- 待办创建时间
- 当前状态

### 4.5 单据状态

单据状态必须是标准化、可追踪状态，不得用自由文本替代正式状态。

当前运行态至少已形成以下展示口径：

- 草稿
- 审批中
- 已通过
- 已驳回
- 流程异常
- 待支付
- 支付中
- 支付完成
- 支付异常

后续若要扩展状态，必须同步前后端枚举、筛选口径、状态标签和测试。

### 4.6 详情导航与导出场景

详情导航是共享详情页的上下文能力，用于在列表与详情之间维持顺序浏览。

导出场景是下载中心异步任务对报销运行态的消费语义，当前至少包括：

- `MY_EXPENSES`
- `PENDING_APPROVAL`
- `DOCUMENT_QUERY`

后续不得让页面各自定义一套与异步任务脱节的导出语义。

## 5. 统一业务规则

### 5.1 共享编辑器规则

- 新建、重提、审批修改必须继续共享 `ExpenseCreateView.vue`
- 页面必须显式区分 `create / resubmit / modify`
- 不允许依赖隐式路由副作用猜测页面模式
- 重提与审批修改不得平行再造第二套填写页

### 5.2 共享详情规则

- 我的报销、待我审批、单据查询进入详情时，统一打开 `/expense/documents/:documentCode`
- 不允许每个列表页各自实现一套详情页
- 详情页中的轨迹、导航、评论、催办、费用明细只读展示必须保持共享口径

### 5.3 列表统一规则

- 我的报销、待我审批、单据查询属于同一报销工作台体系
- 三个列表页必须继续复用共享列定义、状态口径和筛选 helper
- 不允许 3 个页面各自发散出不同状态标签、不同字段命名或不同筛选语义

### 5.4 超时与失败兜底规则

- 模板详情、单据详情、导航、费用明细等运行态主请求必须保留超时与失败兜底
- 不允许整页永久 loading
- 不允许以后端不返回替代前端失败提示
- 前端不得通过猜测默认值掩盖后端真实异常

### 5.5 防跑偏清单

后续每次改动报销管理前，至少先逐项确认：

- 是否仍在 4 个子模块既有路由和接口域内演进
- 是否仍复用现有共享编辑器、共享详情页、共享列表 helper
- 是否所有用户可见中文、状态文案、按钮标题都经过 UTF-8 与实际显示检查
- 是否详情、审批、重提、导出、导航链路都补了对应回归测试
- 是否影响流程管理、付款单、凭证生成等下游模块，若影响则已同步评估

## 6. 前端实施规范

### 6.1 页面与路由规范

- 主路由基线仍在 `frontend/admin-web/src/router/index.ts`
- 4 个子模块分别由 `ExpenseCreateView.vue`、`ExpenseListView.vue`、`ExpenseApprovalView.vue`、`ExpenseDocumentsView.vue` 承载
- 详情页继续由 `ExpenseDocumentDetailView.vue` 承载
- 与费用明细相关的编辑态和只读态继续使用既有 detail 子路由，不新增平行明细页

### 6.2 文案与编码规范

- 所有用户可见文案必须使用 UTF-8 保存，并保证页面实际显示为可读中文
- 禁止提交乱码字符串、错误编码字符串、复制后未修正的 mojibake 文案
- 路由 `meta.title / tabTitle / description` 必须与实际页面语义一致
- 当前 `单据查询` 路由元信息存在乱码，属于不合规示例；后续凡触达该路由必须优先修正

### 6.3 详情与回退规范

- 各列表页进入详情的行为必须保持一致
- 回退逻辑必须优先回到来源语义对应的页面，不得出现错回退、串页或落回无关模块
- 共享详情页上的导航、评论、催办、审批动作矩阵必须基于统一详情模型，不得各处拼装

### 6.4 组件与 helper 复用规范

- 列表页优先复用 `expenseWorkbenchListHelper.ts`
- 运行态表单优先复用 `ExpenseRuntimeFormEditor` 与只读渲染器
- 金额格式化、状态标签、导出场景、下载中心联动应继续复用现有工具和类型
- 不允许无必要地为“我的报销 / 待我审批 / 单据查询”各自新造一套表格体系

### 6.5 前端测试规范

后续触达报销管理前端能力时，至少应按变更范围补齐或回归以下测试基线：

- `ExpenseCreateView.test.ts`
- `ExpenseListView.test.ts`
- `ExpenseApprovalView.test.ts`
- `ExpenseDocumentsView.test.ts`
- `ExpenseDocumentDetailView.test.ts`
- `ExpenseRuntimeFormEditor.test.ts`
- `expenseWorkbenchListHelper.test.ts`

新增功能若改变路由、标题、详情链路、返回行为、状态口径或超时处理，必须补对应断言，避免再次出现乱码、错路由、串页或永久 loading。

## 7. 后端实施规范

### 7.1 分层规范

- `ExpenseDocumentController` 负责新建报销链路的 HTTP 边界
- `ExpenseDocumentQueryController` 负责单据列表、详情、导航、评论、催办、重提等查询与操作边界
- `ExpenseApprovalController` 负责待我审批动作边界
- 运行态核心聚合服务继续以 `ExpenseDocumentService` 为中心
- 不允许在 controller 中直接拼装复杂业务逻辑或直接操作数据库 mapper

### 7.2 接口域规范

- 新建报销相关接口继续收敛在 `/auth/expenses/create/**`
- 我的报销与单据详情相关接口继续收敛在 `/auth/expenses/**`
- 待我审批相关接口继续收敛在 `/auth/expense-approval/**`
- 单据查询继续通过 `/auth/expenses/query-documents`
- 后续不得围绕相同语义再造平行接口域

### 7.3 异常与返回规范

- 详情、审批、修改、重提、评论、催办、导航等动作必须统一走标准失败返回
- 不允许吞异常、悬空无响应、半结构化响应或长时间 pending
- 高风险动作必须返回稳定且可理解的错误结果，供前端稳定提示

### 7.4 DTO / VO 与访问规则规范

- DTO/VO、状态枚举、权限码、列表字段口径变更必须同步前端类型、后端测试和规范
- 详情访问继续按当前跨视图规则处理，不得随意放开或绕开权限
- 审批动作、修改上下文、重提上下文必须继续由 `ExpenseDocumentService` 统一编排，不允许拆成语义重叠的第二套服务

## 8. 数据与 SQL 规范

### 8.1 变更方式规范

- 报销管理相关表结构或种子数据变更只允许通过增量 SQL 落地
- 禁止把整库初始化脚本当成运行态功能发布手段
- 禁止依赖“重建数据库”解决报销链路线上问题

### 8.2 编码与字符集规范

- 数据库层新增表、字段、脚本统一按 UTF-8 / `utf8mb4` 口径处理
- 单据标题、模板名称、费用明细标题、审批意见、轨迹名称等中文数据必须确保可稳定存取
- SQL 文件本身必须为 UTF-8，避免中文主数据写坏

### 8.3 数据兼容规范

- 增量 SQL 必须可在现有库上执行，不得要求整库初始化
- 涉及状态、默认值、模板关联或草稿数据结构调整时，必须考虑历史单据兼容
- 任何可能影响历史详情读取或编辑上下文的变更，都必须附带明确兼容策略

## 9. 权限与审计规范

### 9.1 权限语义

报销管理后续至少应规范以下动作权限：

- 查看并进入新建报销
- 保存与提交新建报销
- 查看我的报销
- 编辑、删除、重提本人单据
- 查看待我审批与执行审批动作
- 查看单据查询与共享详情

本规范只锁定权限语义边界和命名空间，不提前展开另一套新的权限树设计稿。

### 9.2 审计要求

以下动作默认视为高风险动作，必须留痕：

- 提交单据
- 重提单据
- 审批通过 / 驳回
- 审批修改
- 转交 / 加签
- 评论 / 催办

审计信息至少应记录：

- 操作者
- 操作时间
- 目标单据编码
- 所属模块场景
- 关键输入摘要
- 操作结果
- 失败原因

## 10. 测试与验收规范

### 10.1 前端最小回归面

前端后续改动报销管理时，至少应验证以下场景：

- `新建报销` 模板选择、模板详情加载、填写、费用明细编辑、保存草稿、提交
- `我的报销` 列表筛选、详情打开、导出
- `待我审批` 列表展示、查看、通过、驳回
- `单据查询` 列表查询、详情打开、状态展示
- 统一详情页成功或失败都能稳定结束 loading
- 中文标题、tab 标题、按钮文案、空态文案在页面实际可读

### 10.2 后端最小回归面

后端后续改动报销管理时，至少应回归以下测试基线：

- `ExpenseDocumentControllerTest.java`
- `ExpenseDocumentQueryControllerTest.java`
- `ExpenseDocumentServiceImplTest.java`

若本次改动触达新建、详情、审批、重提、导航、导出场景或状态口径，必须补充对应 controller 或 service 测试，防止接口挂起、字段不一致或权限漏失。

### 10.3 人工验收路径

每次较大改动后，至少应按以下路径做人工回归：

- 从左侧菜单分别进入“新建报销 / 我的报销 / 待我审批 / 单据查询”
- 确认四个页面标题、列表、统计卡、按钮文案可读且不乱码
- 从三个列表页进入统一详情页，确认不会串页、不会永久 loading
- 新建一张真实报销单并完成提交、查看、审批、查询回看闭环

## 11. 与现有系统关系

### 11.1 与流程管理的关系

- 流程管理负责提供模板、表单、费用明细设计和费用类型配置
- 报销管理负责消费这些配置完成运行态提单、审批和查询
- 报销管理不得自行维护一套与流程管理脱节的模板定义

### 11.2 与付款单和凭证生成的关系

- 付款单、支付状态、凭证生成属于报销管理的下游消费链路
- 报销管理负责提供正确的单据状态、金额和上下文
- 报销管理不得在运行态页面中直接承担付款执行或总账职责

### 11.3 与下载中心和消息通知的关系

- 下载中心消费报销导出场景，但不改报销管理的页面定义
- 消息通知可承接异步任务消息，但不改变报销管理主链路
- 报销管理页面不应为了下载或通知再造第二套异步状态模型

### 11.4 与基础档案的关系

- 供应商、收款单位、收款人、部门、员工等基础数据来自既有档案或个人中心能力
- 报销管理负责消费这些数据，不负责另造主数据平台

## 12. 后续实施约束 / 引用方式

后续任何报销管理相关页面、接口、权限、SQL、测试方案，都必须先对照本规范。

统一实施顺序如下：

- 先对照规范确认是否在边界内
- 再拆页面、接口、数据结构、权限和测试
- 最后进入开发、联调与验收

引用要求如下：

- 需求评审时，必须注明所对应的规范章节
- 页面方案、接口方案、权限方案、SQL 方案、测试方案必须逐项对照本规范
- 若需求与本规范冲突，必须先修订规范，再进入实现
- 未引用本规范的报销管理相关实现，默认视为规格不完整
