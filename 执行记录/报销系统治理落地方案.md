# 报销系统治理落地方案

本文档是报销系统后续治理、评审、实施、联调与验收的统一基线。当前基线更新时间：2026-04-11。

## 1. 治理目标

- 总目标固定为：先止住“继续长屎山”，再逐步把高风险大文件和聚合热点拆成稳定边界
- 本轮治理明确不做：激进重命名、不先拆微服务、不一次性推倒重来
- 默认交付方式为长期演进：每一项都允许独立分批上线，不要求一次做完

## 3. 总体治理原则

1. 先止血，再瘦身，再收口
2. 所有治理按“先抽边界、再迁移调用、最后收口旧入口”的三段式推进
3. 新功能禁止继续堆进旧热点文件
4. 对外契约默认保持稳定：不改 URL、不改核心接口签名、不改前端 `@/api` 导入方式
5. 每一批都必须可独立回滚、独立上线、独立验收

## 6. 批次实施与验收规则

每一批固定按以下顺序执行：

1. 抽边界
2. 迁移调用
3. 收口旧入口
4. 定向测试
5. façade / 契约回归
6. 全量回归

每一批至少要满足：

- 编译通过
- 定向测试通过
- 高风险主链路 smoke test 通过
- 文档与脚本口径同步

明确禁止：

- 带着编译红灯进入下一批
- 带着主链路断裂进入下一批
- 以“后面再补”为理由继续往热点文件堆新逻辑

## 7. 对外契约与内部演进约束

### 7.1 默认不动的对外契约

- controller URL
- 既有 service 接口签名
- 前端 `@/api` 导入路径
- 核心 DTO / VO 字段名
- money 字段表示方式

### 7.2 允许的内部演进方式

- 新增 domain service / support / assembler / module file
- 旧入口先兼容委派，再逐批收口
- 新增 owner 必须有清晰边界，禁止桥接层搬家后继续长成新屎山

### 7.3 禁止的内部演进方式

- 激进重命名
- 未稳定前先拆微服务
- 直接推翻现有 working facade / barrel / route 结构重写
