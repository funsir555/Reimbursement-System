# 报销系统治理落地方案

本文档是报销系统后续治理、评审、实施、联调、验收的统一基线；后续涉及报销架构优化时，应优先引用本文档，不得临时改口径。

更新日期：2026-04-09

## 1. 治理目标

- 总目标固定为：先止住“继续长屎山”，再逐步把高风险大文件拆成稳定边界。
- 本轮治理明确不做：激进重命名、不先拆微服务、不一次性推倒重来。
- 默认交付方式为长期演进：每一项都允许独立分批上线，不要求一次做完。

## 2. 当前治理进度基线

### 2.1 已取得的进展

- `ExpenseDocumentServiceImpl` 已从超大实现收口到 façade 方向。
- `frontend/admin-web/src/api/index.ts` 已基本收成兼容层 / barrel。
- 配置与启动契约已开始收紧，`start-finex.bat`、README、环境变量示例已有初步护栏。

### 2.2 当前仍需重点治理的热点

- `backend/auth-service/src/main/java/com/finex/auth/service/impl/expense/ExpenseDocumentMutationSupport.java`
- `backend/auth-service/src/main/java/com/finex/auth/service/impl/expense/ExpenseWorkflowRuntimeSupport.java`
- `frontend/admin-web/src/api/modules/shared.ts`
- `frontend/admin-web/src/router/index.ts`

### 2.3 当前阶段判断

- 项目已从“完全失控的大类堆积”进入“已有初步边界，但残余热点仍有失控风险”的阶段。

## 3. 总体治理原则

1. 先止血，再瘦身，再收口。
2. 所有治理按“先抽边界、再迁移调用、最后收口旧入口”的三段式推进。
3. 新功能禁止继续堆进旧热点文件。
4. 对外契约默认保持稳定：不改 URL、不改核心接口签名、不改前端 `@/api` 导入方式。
5. 每一批都必须可独立回滚、独立上线、独立验收。

## 4. 治理顺序与落地拆分

治理顺序固定按“风险最高、收益最大、牵引面最广”排序，不得交换。

### 4.1 ExpenseDocumentServiceImpl

- 目标：继续收口报销运行态核心链路，防止 façade 背后的残余大类继续膨胀。
- 当前主要问题：`ExpenseDocumentMutationSupport.java` 仍承载过多业务残余；runtime、submission、query、approval、payment 的边界虽已开始拆，但仍未完全稳定。
- 三段式推进：
  1. 先抽边界：继续识别只属于 submission / query / approval / payment / runtime 的 helper 与状态推进逻辑。
  2. 再迁移调用：让 façade 和各 domain service 直接依赖真正 owner，不再经过 mutation support 兜转。
  3. 最后收口旧入口：`ExpenseDocumentMutationSupport` 只保留极小残余 helper，不再承接 live 主流程。
- 批次验收：compile 通过、focused tests 通过、`ExpenseDocumentServiceImplTest` 通过、报销主链路 smoke test 可走通。

### 4.2 前端 API 中心

- 目标：从“`index.ts` 已 barrel 化”继续推进到“共享类型与分域模块真正稳定”。
- 当前主要问题：`src/api/modules/shared.ts` 仍然过大，跨域类型继续堆积，新的共享热点已经形成。
- 三段式推进：
  1. 先抽边界：把仅属于某个域的类型从 `shared.ts` 下沉回各自模块。
  2. 再迁移调用：页面、store、test 逐步改为从分域模块消费类型与 API。
  3. 最后收口旧入口：`index.ts` 继续只做兼容导出，`shared.ts` 只保留真正跨域公共定义。
- 批次验收：`@/api` 兼容导入不破、API 模块测试通过、前端 build 和 unit test 保持全绿。

### 4.3 路由 / 菜单 / 权限配置统一

- 目标：把路由定义、菜单展示、权限判断、fallback 口径统一为同一套前端真相源。
- 当前主要问题：`src/router/index.ts` 仍然过大，meta、权限码、fallback、菜单语义存在耦合。
- 三段式推进：
  1. 先抽边界：抽出 route meta、menu config、permission capability 的独立配置层。
  2. 再迁移调用：router guard、MainLayout、权限 util、菜单渲染统一消费同一份配置。
  3. 最后收口旧入口：`router/index.ts` 只保留路由注册与守卫编排。
- 批次验收：登录跳转、菜单展示、权限拦截、fallback 路径、关键页面进入行为全部保持一致。

### 4.4 auth-service 单体内领域收口

- 目标：在不拆微服务前提下，把 `auth-service` 内部领域边界继续收紧。
- 当前主要问题：虽然已有 `auth-service领域边界说明.md`，但热点实现和 support 仍有跨域堆积。
- 三段式推进：
  1. 先抽边界：按 `auth / profile / process / async-task / voucher / settings / mvp-dashboard` 识别现有归属。
  2. 再迁移调用：把只属于单一领域的 controller / service / support / helper 回归所属域。
  3. 最后收口旧入口：大 controller 与大 service 进入冻结期，只保留兼容委派。
- 批次验收：领域边界文档同步更新，新增代码不再反向堆回大类，主要 controller / service 回归测试通过。

### 4.5 配置与环境契约收紧

- 目标：把配置、环境变量、启动脚本、运行时校验统一成明确契约，避免隐式默认值继续埋雷。
- 当前主要问题：虽然数据库 / JWT / 启动脚本已有护栏，但不同入口仍可能存在不完全一致的配置口径。
- 三段式推进：
  1. 先抽边界：统一列出必须显式声明的配置项与允许保留的安全默认值。
  2. 再迁移调用：让 README、示例环境文件、启动脚本、运行时校验器、`application.yml` 对齐同一套契约。
  3. 最后收口旧入口：去掉危险默认值与静默 fallback，缺失关键变量时明确失败。
- 批次验收：启动脚本、后端启动、前端联调、README 与环境示例口径一致。

## 5. 风险控制与备份策略

- 凡涉及热点大文件、跨模块迁移、配置入口、启动脚本或公共契约调整，先备份再实施。
- 备份目录固定放桌面，使用时间戳命名，例如 `C:\Users\funsir\Desktop\报销系统-backup-YYYY-MM-DD-HHMMSS`。
- 备份方式固定为完整副本，不排除 `.git`、`backend`、`frontend`、`docs`、`dist`、`node_modules`、`target`。
- 建议命令固定如下：

```powershell
robocopy "C:\Users\funsir\Desktop\报销系统" "C:\Users\funsir\Desktop\报销系统-backup-YYYY-MM-DD-HHMMSS" /E /COPY:DAT /R:1 /W:1 /XJ
```

- 返回码 `0-7` 视为成功，`8+` 视为失败；失败则停止进入实施阶段。
- 备份验证至少检查：目标目录存在，且顶层包含 `.git`、`backend`、`frontend`、`docs`。
- 本次文档写入本身不属于高风险代码改动，不需要先做桌面整仓备份；但后续凡进入高风险治理批次，必须先备份再实施。

## 6. 批次实施与验收规则

每一批固定按以下顺序执行：

1. 抽边界
2. 迁移调用
3. 收口旧入口
4. 定向测试
5. façade / 契约回归
6. 全量回归

每一批至少要满足：

- 编译通过
- 定向测试通过
- 高风险主链路 smoke test 通过
- 文档与脚本口径同步

明确禁止：

- 不允许带着编译红灯进入下一批
- 不允许带着主链路断裂进入下一批
- 不允许以“后面再补”为理由继续往热点文件堆新逻辑

## 7. 对外契约与内部演进约束

### 7.1 默认不动的对外契约

- controller URL
- `ExpenseDocumentService` 接口签名
- 前端 `@/api` 导入路径
- 核心 DTO 字段名
- money 字段表示方式

### 7.2 允许的内部演进方式

- 新增 domain service / support / assembler / module file
- 旧入口先兼容委派，再逐批收口

### 7.3 禁止的内部演进方式

- 激进重命名
- 未稳定前先拆微服务
- 直接推翻现有 working façade / barrel / route 结构重写

## 8. 测试与回归口径

后续每次引用本文档做架构优化，都必须先说明当前处于哪一项、哪一批、哪一阶段，不能跳批执行。

验收固定覆盖 4 层：

### 8.1 代码级

- compile
- focused test
- domain / service / controller 回归

### 8.2 服务级

- 启动
- 接口
- 权限
- 配置
- 脚本联动

### 8.3 前端级

- router / menu / permission / API 契约
- build
- unit test

### 8.4 业务级

报销主链路 smoke test 至少覆盖：

- 新建提交
- 审批
- 付款 / 异常
- 详情查询
