## 1. 治理目标

- 总目标固定为：先止住“继续长屎山”，再逐步把高风险大文件拆成稳定边界。
- 本轮治理明确不做：激进重命名、不先拆微服务、不一次性推倒重来。
- 默认交付方式为长期演进：每一项都允许独立分批上线，不要求一次做完。

## 3. 总体治理原则

1. 先止血，再瘦身，再收口。
2. 所有治理按“先抽边界、再迁移调用、最后收口旧入口”的三段式推进。
3. 新功能禁止继续堆进旧热点文件。
4. 对外契约默认保持稳定：不改 URL、不改核心接口签名、不改前端 `@/api` 导入方式。
5. 每一批都必须可独立回滚、独立上线、独立验收。

## 4. 治理顺序与落地拆分

治理顺序固定按“风险最高、收益最大、牵引面最广”排序，不得交换。

### 4.1 ExpenseDocumentServiceImpl

- 目标：继续压缩 façade 背后的 residual hotspot，防止已拆出的 owner 再次回流长成新屎山。

  1. 继续抽掉 `ExpenseDocumentMutationSupport` 中仍只属于 submission / template / detail 的 residual helper。
  2. 继续压 `ExpenseWorkflowRuntimeSupport` 的 runtime 内部边界，但不再回流 use-case 外壳。
  3. 禁止新增 owner 再次长成“桥接层搬家后的新万能 support”。
- 三段式推进仍保持不变：
  1. 先抽边界：识别只属于 submission / template / detail / runtime 的 residual helper。
  2. 再迁移调用：让 façade 和各 domain service 继续只依赖真正 owner。
  3. 最后收口旧入口：`ExpenseDocumentMutationSupport` 只保留极小 residual helper，`ExpenseWorkflowRuntimeSupport` 只保留 runtime kernel。

1. 抽边界
2. 迁移调用
3. 收口旧入口
