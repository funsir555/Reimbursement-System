# 流程管理模块管理规范

更新日期：2026-04-08

## 1. 文档定位

本文档是“管理工作台 > 流程管理”模块的管理规范与实施基线，不是原型说明，不是接口设计稿，不是数据库表结构设计稿，也不是页面操作手册。

后续凡涉及以下主题的设计、评审、开发、联调、测试与验收，均必须先对照本规范：

- 流程管理工作台
- 单据模板管理
- 流程设计器
- 表单设计器
- 费用明细设计器
- 自定义档案管理
- 费用类型管理
- 流程管理相关权限、SQL、测试与发布

当前本文档只规范“流程配置中心”能力，不展开以下运行态能力的真实功能细则：

- 单据详情页
- 审批运行态与待办页
- 支付链路与付款单
- 首页工作台统计
- 下载中心与消息通知

上述能力虽会消费流程管理产出的配置，但不属于本文档的主体实现范围；后续若要调整其产品定义，必须先补专题规范或修订本规范后再实施。

本文档自发布之日起，作为流程管理模块后续工作的唯一实施基线。

## 2. 模块定义

流程管理模块是报销管理域下的配置中心，用于维护单据模板、审批流程、业务表单、报销专用费用明细表单、自定义档案与费用类型等基础配置能力。

流程管理模块必须服务于以下业务场景：

- 为报销单、申请单、借款单、合同单提供统一模板定义
- 为模板绑定审批流程、表单设计和展示规则
- 为报销单提供费用明细设计与费用模式配置
- 为运行态单据提供费用类型、自定义档案等可选配置来源
- 为报销管理工作台提供统一的流程配置入口

流程管理模块当前不负责以下内容：

- 单据提交、审批、驳回、评论、催办等运行态动作
- 单据详情页渲染与审批轨迹展示
- 支付节点执行页、付款单工作台、银企直连
- 报销首页门户统计与运营类看板
- 独立于报销系统之外的第二套流程产品

## 3. 核心口径与边界

### 3.1 模块归属口径

流程管理属于报销管理架构内的配置能力，不是独立产品，不允许脱离报销管理工作台另造第二套流程中心。

即：

- 菜单入口继续挂在“管理工作台”下
- 前端主入口继续是 `/expense/workbench/process-management`
- 页面风格继续沿用报销管理工作台的布局、卡片、列表和设计器交互体系
- 不允许新增一个与报销管理平级、但语义重复的流程产品入口

### 3.2 前端路由口径

当前前端流程管理基线以以下路由链路为准：

- `/expense/workbench/process-management`
- `/expense/workbench/process-management/create/:templateType`
- `/expense/workbench/process-management/edit/:templateType/:id`
- `/expense/workbench/process-management/flow-designer/create`
- `/expense/workbench/process-management/flow-designer/:id`
- `/expense/workbench/process-management/form-designer/create`
- `/expense/workbench/process-management/form-designer/:id`
- `/expense/workbench/process-management/expense-detail-designer/create`
- `/expense/workbench/process-management/expense-detail-designer/:id`

后续新增流程管理能力必须优先在这条既有路由域下演进，不得随意拆出不一致的新入口。

### 3.3 后端接口域口径

流程管理后端统一走现有接口域：

- `/auth/process-management/**`

后续新增模板、流程、表单、费用类型、自定义档案相关能力，必须优先在该域下扩展，不得另造平行控制器域或平行 API 前缀。

### 3.4 权限口径

流程管理权限统一继续沿用 `expense:process_management:*` 命名空间。

当前至少已形成以下权限语义：

- `expense:process_management:view`
- `expense:process_management:create`
- `expense:process_management:edit`
- `expense:process_management:disable`
- `expense:process_management:publish`

后续不得把同一类流程管理能力散落到其它无关权限域。

### 3.5 当前现状基线

当前流程管理模块的真实架构基线按以下实现理解：

- 前端主工作台以 `frontend/admin-web/src/views/process/ProcessManagementView.vue` 为中心
- 前端路由入口定义在 `frontend/admin-web/src/router/index.ts`
- 前端 API 收敛在 `frontend/admin-web/src/api/index.ts` 的 `processApi`
- 后端入口控制器为 `backend/auth-service/src/main/java/com/finex/auth/controller/ProcessManagementController.java`
- 后端聚合编排服务为 `backend/auth-service/src/main/java/com/finex/auth/service/impl/ProcessManagementServiceImpl.java`
- 工作台当前导航统一为“单据与流程 / 费用明细表单 / 自定义档案 / 费用类型”

后续所有扩展都必须建立在这套基线上，不允许忽略现状直接重写一套新体系。

## 4. 核心对象定义

### 4.1 模板类型

模板类型是流程管理的一级业务语义，不得使用自由文本代替。

当前模板类型统一按以下口径处理：

- `report`：报销单
- `application`：申请单
- `loan`：借款单
- `contract`：合同单

后续若要扩展新的模板类型，必须先修订本规范，并同步前后端类型、标签、路由文案和测试。

### 4.2 单据模板

单据模板是流程管理的核心配置对象，用于描述一类单据的业务身份和配置绑定关系。

单据模板至少应具备以下语义：

- 模板编码
- 模板名称
- 模板类型
- 模板分类
- 模板描述
- 适用范围
- 表单设计绑定
- 审批流程绑定
- 费用明细设计绑定
- 默认费用模式
- 启用状态

### 4.3 模板分类与范围

模板分类用于组织模板的工作台展示与业务归类；模板范围用于控制模板可适用的部门、费用类型和其它约束。

模板分类与范围是模板的从属配置，不允许独立演化成与模板脱节的第二套规则中心。

### 4.4 流程设计

流程设计用于维护审批节点、审批人解析规则、流程场景和发布状态。

流程设计必须与模板绑定关系明确，且发布语义稳定：

- 草稿与已发布状态必须可区分
- 模板引用的流程应当指向可用流程
- 流程变更不得绕过发布语义直接影响线上模板

### 4.5 表单设计

表单设计是模板运行态主表单的结构定义，用于决定提单、详情和只读渲染所消费的表单字段集合。

表单设计必须保持以下边界：

- 它服务于模板，不独立承担单据运行态业务逻辑
- 字段结构调整必须同步考虑运行态表单编辑器和只读渲染器
- 变更后不得让既有模板进入无法渲染或无法回显状态

### 4.6 费用明细设计

费用明细设计是报销单专用的子表单设计能力，不是全模板类型通用能力。

当前统一按以下口径处理：

- 只有报销单模板允许绑定费用明细设计
- 申请单、借款单、合同单不得误绑定报销专用费用明细设计
- 默认费用模式只在报销单模板下有效

### 4.7 自定义档案

自定义档案用于维护流程配置中心消费的扩展型业务档案，例如标签、分期付款等业务配置项。

自定义档案必须服从流程管理域的统一规则，不得演化成脱离流程管理的通用主数据平台。

### 4.8 费用类型

费用类型是报销管理运行态消费的重要基础配置，流程管理只负责维护其树结构、展示口径与相关配置。

流程管理中的费用类型不负责承担单据详情、审批轨迹、支付状态等运行态页面职责。

## 5. 统一业务规则

### 5.1 模板绑定统一规则

- 单据模板必须显式绑定流程、表单和相关基础配置，不允许靠前端硬编码兜底
- 报销单模板必须绑定费用明细设计
- 申请单、借款单、合同单不得绑定费用明细设计或默认费用模式
- 模板类型、模板描述、模板分类和适用范围必须保持前后端语义一致

### 5.2 发布与启停统一规则

- 新建、修改、发布、删除、启停必须是受控动作
- 不允许通过直接改数据库或跳过服务层的方式改变模板、流程、表单状态
- 发布动作必须有明确校验，不允许把明显不完整的配置当成正式可用配置
- 删除动作必须是安全删除语义，不得破坏线上已引用配置的数据可读性

### 5.3 工作台统一规则

- 流程管理工作台只负责配置总览与配置入口，不承担运行态审批页职责
- 工作台卡片、侧边导航、模板分组与设计器入口必须保持一致的产品语义
- 不允许把凭证生成、支付执行、下载中心等无关页面挂到流程管理路由下

### 5.4 兼容性统一规则

- 已有模板、流程、表单、费用类型、自定义档案数据在升级后必须可继续读取
- DTO/VO 或枚举扩展时，必须保证旧数据至少可被兼容展示和解释
- 任何需要破坏兼容性的调整，必须先修订规范、给出迁移方案并补测试

### 5.5 防跑偏清单

后续每次改动流程管理前，至少先逐项确认：

- 是否仍在报销管理架构内实现，而不是新增第二入口、第二接口域或第二权限体系
- 是否菜单、路由、tab 标题、页面组件挂载关系仍然一致
- 是否所有用户可见文案都经过中文可读性和 UTF-8 编码检查
- 是否后端接口仍走 `/auth/process-management/**`
- 是否新增或修改能力已经同步补齐前端测试、后端测试与人工回归路径

## 6. 前端实施规范

### 6.1 路由与菜单规范

- 流程管理菜单继续归属“管理工作台”，不得挂到支付、财务管理或其它无关父菜单
- `frontend/admin-web/src/router/index.ts` 是流程管理路由元信息的唯一基线
- 路由 `meta.title`、`meta.tabTitle`、`description` 必须与实际页面语义一致
- 设计器与模板页必须继续作为流程管理子链路，不得误指向凭证生成或其它无关组件

### 6.2 文案与编码规范

- 所有用户可见文案必须使用 UTF-8 保存，并保证页面实际显示为可读中文
- 禁止提交乱码字符串、错误编码字符串、复制后未修正的 mojibake 文案
- 禁止长期保留英文占位标题替代正式中文标题
- 当前若存在英文 `meta.title` 等历史遗留，后续凡触达该路由必须顺手修正为中文并同步测试

### 6.3 页面结构与组件复用规范

- 流程管理工作台优先复用报销管理工作台现有布局、卡片、表格、抽屉、对话框、设计器 helper
- 不允许无必要地再造一套平行状态管理、列表样式或设计器交互
- 新增能力优先复用 `processApi`、现有类型模型和既有设计器辅助函数
- 同一业务语义不得在多个页面使用不同标题、不同按钮名或不同字段口径

### 6.4 类型与接口对齐规范

- 任何后端 DTO/VO 变更都必须同步前端 `processApi` 类型、页面使用点和测试桩
- 模板类型、流程状态、设计器字段类型等枚举不得在前端自行发散
- 页面不得依赖“缺字段时前端猜测默认值”作为正式实现

### 6.5 前端测试规范

后续触达流程管理前端能力时，至少应按变更范围补齐或回归以下测试基线：

- `ProcessManagementView.test.ts`
- `ProcessFlowDesignerView.test.ts`
- `ProcessFormDesignerView.test.ts`
- `ProcessTemplateCreateView.test.ts`
- `processFlowDesignerHelper.test.ts`
- `formDesignerHelper.test.ts`
- `flowDesignerPersistence.test.ts`

新增功能若改变工作台路由、标题、设计器行为或模板绑定逻辑，必须补对应断言，避免再次出现乱码、错路由或串页问题。

## 7. 后端实施规范

### 7.1 分层规范

- `ProcessManagementController` 只负责权限校验、入参绑定、结果包装和 HTTP 边界
- 聚合编排继续由 `ProcessManagementServiceImpl` 负责
- 流程、表单、费用明细、自定义档案、费用类型等细分逻辑继续留在各自专门服务中
- 不允许在 controller 中直接拼装复杂业务逻辑或直接操作数据库 mapper

### 7.2 接口与异常规范

- 所有流程管理接口继续统一返回标准结果结构
- 任何异常都必须进入统一失败返回，不允许吞异常、半成功、半结构化响应或长期 pending
- 不允许以后端无响应来替代前端失败提示
- 保存、发布、删除、解析等动作必须给出稳定且可理解的错误信息

### 7.3 DTO / VO 与服务边界规范

- DTO/VO 字段变更必须同步 controller、service、前端类型、测试和文档
- 模板类型扩展、流程节点语义调整、设计器字段语义调整都属于高风险变更，不允许只改单侧
- 聚合服务可以组合 `processFlowDesignService`、`processFormDesignService`、`processExpenseDetailDesignService` 等专门服务，但不得把这些边界重新揉成不可维护的大一统实现

### 7.4 保存与发布规范

- 新建、修改、删除、启停、发布都必须保留服务层校验和审计留痕能力
- 不允许绕过服务层直接通过 SQL 或 Mapper 拼装写入以图省事
- 对运行态有影响的模板、流程、表单变更，必须确保线上引用对象仍可正确读取

## 8. 数据与 SQL 规范

### 8.1 变更方式规范

- 流程管理相关表结构或种子数据变更只允许通过增量 SQL 落地
- 禁止把整库初始化脚本当成日常功能发布手段
- 禁止依赖“重建数据库”解决流程管理线上问题

### 8.2 编码与字符集规范

- 数据库层新增表、字段、脚本统一按 UTF-8 / `utf8mb4` 口径处理
- 枚举标签、模板标题、流程名称、设计器字段名称等中文数据必须确保可稳定存取
- SQL 文件本身必须为 UTF-8，避免因为脚本编码问题把中文主数据写坏

### 8.3 数据兼容规范

- 增量 SQL 必须可在现有库上执行，不得要求整库初始化
- 涉及模板类型、状态、默认值、种子数据调整时，必须考虑历史数据兼容
- 任何可能影响历史模板读取的变更，都必须附带明确迁移或兼容策略

## 9. 权限与审计规范

### 9.1 权限语义

流程管理后续至少应规范以下动作权限：

- 查看工作台和配置详情
- 创建模板 / 流程 / 表单 / 费用明细设计 / 档案 / 费用类型
- 修改既有配置
- 启停既有配置
- 发布流程或模板

本规范只锁定权限语义边界和命名空间，不提前展开另一套新的权限树设计稿。

### 9.2 审计要求

以下动作默认视为高风险动作，必须留痕：

- 新建模板、流程、表单、费用明细设计
- 修改模板、流程、表单、费用类型、自定义档案
- 删除配置对象
- 发布流程
- 启停配置

审计信息至少应记录：

- 操作者
- 操作时间
- 操作对象类型
- 目标对象主键或编码
- 关键输入摘要
- 操作结果
- 失败原因

## 10. 测试与验收规范

### 10.1 前端最小回归面

前端后续改动流程管理时，至少应验证以下场景：

- “管理工作台 > 流程管理”入口可见且能打开正确页面
- 工作台概览、侧边导航、模板列表、流程入口、表单入口可正常加载
- 新建 / 编辑模板页不串页、不误挂、标题不乱码
- 流程设计器、表单设计器、费用明细设计器的关键保存与回显链路正常
- 中文标题、tab 标题、按钮文案、空态文案在页面实际可读

### 10.2 后端最小回归面

后端后续改动流程管理时，至少应回归以下测试基线：

- `ProcessManagementControllerTest.java`
- `ProcessManagementServiceImplTest.java`
- `ProcessFlowDesignServiceImplTest.java`
- `ProcessExpenseDetailDesignServiceImplTest.java`

若本次改动触达模板保存、流程发布、表单设计、费用类型、自定义档案等能力，必须补充对应 controller 或 service 测试，防止接口挂起、字段不一致或校验漏失。

### 10.3 人工验收路径

每次较大改动后，至少应按以下路径做人工回归：

- 从左侧菜单进入“管理工作台 > 流程管理”
- 查看工作台 4 个导航区块是否与预期一致
- 新建一个模板并检查模板类型、流程、表单、费用明细绑定是否符合规范
- 进入流程设计器、表单设计器、费用明细设计器确认标题、保存、回显正常
- 检查自定义档案、费用类型是否仍可正常查询与维护

## 11. 与现有系统关系

### 11.1 与报销创建的关系

- 报销创建页消费流程管理产出的模板、表单、费用明细设计和费用类型
- 流程管理负责配置来源，报销创建负责运行态消费
- 流程管理不得在配置页直接承担提单页职责

### 11.2 与单据详情和审批运行态的关系

- 单据详情页和审批运行态负责展示、审批、轨迹和评论等运行态行为
- 流程管理负责这些运行态所依赖的配置，不负责直接承接运行态操作页
- 若运行态因配置变更需要调整，必须同步评估详情页与审批页兼容性

### 11.3 与费用类型、自定义档案和基础档案的关系

- 费用类型和自定义档案在流程管理域内维护，但会被报销创建和运行态消费
- 供应商、客户、员工、部门等基础档案仍由各自档案模块负责
- 流程管理不得自行复制一套与基础档案脱节的主数据定义

### 11.4 与测试体系的关系

- 流程管理的前端单测、后端控制器测试、后端服务测试共同构成最小质量防线
- 任何声称“只是小改动”的变更，只要影响用户可见文案、路由、接口或绑定关系，都必须进入回归范围

## 12. 后续实施约束 / 引用方式

后续任何流程管理相关页面、接口、权限、SQL、测试方案，都必须先对照本规范。

统一实施顺序如下：

- 先对照规范确认是否在边界内
- 再拆页面、接口、数据结构、权限和测试
- 最后进入开发、联调与验收

引用要求如下：

- 需求评审时，必须注明所对应的规范章节
- 页面方案、接口方案、权限方案、SQL 方案、测试方案必须逐项对照本规范
- 若需求与本规范冲突，必须先修订规范，再进入实现
- 未引用本规范的流程管理相关实现，默认视为规格不完整

## 数据库字段标准补充

- 涉及数据库建表、加列、改列时，所有表注释和字段注释必须使用真实中文，禁止 `column comment`、`table comment`、纯英文占位和 `?` 乱码。
- 金额字段统一使用 `DECIMAL(18,2)` 作为货币类型标准；费率、汇率、数量、工作量等非金额字段不得误套该规则。
- 新库只执行 `init*.sql`，旧库升级只执行 `migrate*.sql`，禁止手工点改数据库后不留脚本。
