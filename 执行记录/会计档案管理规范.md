## 2. 模块定义

会计档案是财务域内部的基础主数据中心，用于承载公司级财务核算对象、往来对象、组织归属对象、项目核算对象和总账核心科目对象。

会计档案模块必须服务于以下财务场景：

- 凭证录入与修改
- 总账查询与辅助核算选择
- 客户 / 供应商 / 员工 / 部门 / 项目相关核算口径
- 账套初始化后的基础档案维护
- 后续导入、导出、批量维护与审计追踪

会计档案模块当前不负责以下内容：

- 日常凭证审批与记账动作
- 报销业务单据流转
- 财务账套生命周期管理
- 系统级组织架构维护的产品定义

## 3. 核心口径与边界

### 3.1 公司隔离口径

本项目的会计档案统一受“财务上下文 + 档案隔离口径”双重约束。

即：

- 客户、供应商、会计科目、项目档案必须跟随当前财务公司上下文，列表查询、详情、创建、编辑、停用、导入、导出都必须按当前公司隔离
- 员工、部门档案自本规范修订起改为财务域全局共享基础档案，不按公司隔离，不得再把“归属公司”作为前端筛选或后端过滤前提
- 不允许在会计档案页内再造一套与财务上下文冲突的独立公司状态
- 不允许把共享员工、共享部门错误实现成“随公司切换而切源”的伪共享口径
- 不允许跨公司读取、编辑或停用其他公司的客户、供应商、会计科目、项目档案

### 3.2 页面形态口径

会计档案模块的读写型页面默认沿用当前财务域已落地的主档维护模式。

即：

- 紧凑页头
- 顶部筛选区
- 下方列表区
- 弹窗或抽屉维护明细
- 不默认采用跳转式表单页

客户档案、供应商档案已经形成首批参考口径；后续部门档案和会计科目档案应优先按同一交互密度和信息分层落地。

### 3.3 模块独立口径

会计档案模块独立隶属于“财务管理”，不得与“系统设置”或“电子档案”混同。

即：

- 产品定义独立
- 页面职责独立
- 权限语义独立
- 财务上下文独立

后续若底层存在共用表、共用同步逻辑或共用人员 / 部门数据源，只允许在实现层共用，不得在产品规范层混并入口。

### 3.4 当前现状口径

当前财务架构下，会计档案模块按以下现状理解并约束后续实现：

- 客户档案：已是财务域真实维护页，按公司隔离，支持查询、创建、编辑、停用
- 供应商档案：已是财务域真实维护页，按公司隔离，支持查询、创建、编辑、停用
- 员工档案：当前属于财务域查询型实现，主要用于人员核算与归属展示；后续继续按财务全局共享档案落地
- 部门档案：已落地为财务域真实只读页，按财务全局共享档案口径展示；维护入口统一保留在系统设置 > 组织架构
- 项目档案：当前仍属静态占位，后续用于承接总账项目分类 / 项目辅助核算正式来源
- 会计科目：当前仍属占位或待真实化能力，但已是后续总账和凭证能力的核心依赖对象

后续真实化时，不得破坏客户 / 供应商已形成的财务域实现模式。

## 4. 核心对象定义

### 4.1 客户档案

客户档案是公司级客户往来主数据对象，用于承载应收、开票、结算、联系人与客户扩展属性。

客户档案必须至少具备以下语义：

- 归属公司
- 客户编码
- 客户名称
- 联系与银行信息
- 往来余额相关信息
- 状态与停用信息
- 更新时间与审计信息

### 4.2 供应商档案

供应商档案是公司级供应商往来主数据对象，用于承载应付、付款、收款账户、税务与供应商扩展属性。

供应商档案必须至少具备以下语义：

- 归属公司
- 供应商编码
- 供应商名称
- 联系与收款信息
- 往来余额相关信息
- 状态与停用信息
- 更新时间与审计信息

### 4.3 员工档案

员工档案是财务域全局共享的人员核算与归属对象，用于承载凭证辅助核算、个人往来、组织归属、角色归属和同步来源信息。

员工档案必须至少具备以下语义：

- 人员身份标识
- 所属部门
- 劳动关系归属
- 来源类型
- 状态
- 最近同步信息

### 4.4 部门档案

部门档案是财务域全局共享的组织核算对象，用于承载部门辅助核算、组织归属、负责人和层级结构。

财务管理 > 会计档案 > 部门档案首版只提供查看入口，不提供新增、编辑、删除、导入、导出等维护动作；组织架构维护统一在系统设置 > 组织架构完成。

部门档案必须至少具备以下语义：

- 部门编码
- 部门名称
- 父子层级关系
- 负责人
- 状态
- 同步来源与同步状态

### 4.5 会计科目档案

会计科目档案是会计档案模块中的首要核心对象，是总账、凭证、辅助核算、账簿查询和账套后续维护的主数据基础。

会计科目必须至少具备以下语义：

- 归属公司
- 科目编码
- 科目名称
- 科目层级
- 科目性质与类别
- 末级状态
- 启用 / 封存状态
- 辅助核算挂载能力
- 现金 / 银行 / 对账类属性
- 扩展定义能力

### 4.6 项目档案

项目档案是公司级项目核算对象，用于承载项目分类、项目主数据、启停状态和与凭证辅助核算的关联能力。

项目档案必须至少具备以下语义：

- 归属公司
- 项目分类
- 项目编码
- 项目名称
- 状态
- 审计信息

## 5. 模块级统一业务规则

### 5.1 查询与维护规则

- 所有会计档案页面必须先完成财务上下文初始化，再执行查询
- 列表页必须至少支持关键词查询、状态筛选、刷新、重置
- 明细维护优先采用弹窗或抽屉，不默认拆分成独立跳转页
- 客户、供应商、会计科目、项目档案编辑时必须校验当前对象与当前公司一致
- 员工、部门档案编辑时必须校验对象处于全局共享可用状态，不得误套公司隔离校验
- 若财务域页面当前仅提供员工、部门档案只读查看，则不得在财务页内补做绕开系统设置的维护入口
- 停用必须是受控动作，不得直接无痕物理删除作为默认口径

### 5.2 生命周期与状态规则

所有会计档案对象必须采用受控状态，不得用无法检索的自由文本状态代替。

首版至少应支持以下语义：

- 启用
- 停用 / 封存
- 可查询状态
- 可审计的更新时间

后续如需支持草稿、待审核、同步中等扩展状态，必须先修订规范再落地。

### 5.3 导入 / 导出 / 批量操作规则

以下动作默认视为重操作：

- 导入
- 导出
- 批量启用
- 批量停用
- 批量重建编码
- 批量调整辅助核算属性

上述动作默认不得按长同步请求直接实现，必须预留任务化与结果留痕口径。

### 5.4 与总账联动规则

- 会计档案是总账的基础数据源，不得与总账元数据口径脱节
- 会计科目、客户、供应商、员工、部门、项目作为辅助核算对象时，必须与凭证编辑器口径一致
- 若档案对象已停用，后续在凭证页的选择、引用、保存策略必须先按规范定义后再实现
- 未完成真实化的档案对象，不得长期依赖硬编码种子作为正式口径

### 5.5 项目档案过渡规则

- 项目档案在首版真实化前允许以静态占位页面存在
- 静态占位不等于产品定义缺失，后续真实化必须严格回到本规范
- 项目档案后续主要服务于总账凭证中的 `项目分类 / 项目` 辅助核算口径
- 在项目档案未真实化前，不得长期继续把硬编码项目种子视为正式主数据来源

## 6. 会计科目重点规范

### 6.1 产品定位

会计科目是会计档案模块中的首要核心对象。

后续以下能力都必须以会计科目档案为前置基础：

- 新建凭证
- 查询凭证
- 凭证详情与修改
- 余额类查询
- 明细账与辅助账查询
- 账套初始化后的科目维护

会计科目不得被视为普通静态字典，也不得继续长期依赖前端或后端硬编码种子。

### 6.2 字段分组口径

基于《会计科目档案-字段信息表》，会计科目首版字段规则统一按以下 6 组理解：

#### 6.2.1 编码与层级

首批核心字段包括：

- `ccode`：科目编码
- `ccode_name`：科目名称
- `igrade`：编码级次
- `chelp`：科目助记码

该组字段必须用于支撑：

- 科目树或分级展示
- 末级判断
- 编码唯一性校验
- 快速检索

#### 6.2.2 科目属性

首批核心字段包括：

- `cclass`：科目类型
- `cclassany`：财务分析类型
- `bproperty`：科目性质
- `cbook_type`：账页格式

该组字段必须用于支撑：

- 资产 / 负债 / 权益 / 成本 / 损益等口径归类
- 后续账页或报表取数适配
- 科目属性展示与受控维护

#### 6.2.3 辅助核算

首批核心字段包括：

- `bperson`
- `bcus`
- `bsup`
- `bdept`
- `bitem`
- `cass_item`

该组字段必须用于支撑：

- 个人往来核算
- 客户往来核算
- 供应商往来核算
- 部门核算
- 项目核算
- 项目大类挂载

辅助核算标识一旦被实际业务引用，后续变更必须按受控变更处理，不得随意修改。

#### 6.2.4 现金银行与对账

首批核心字段包括：

- `bcash`
- `bbank`
- `bused`
- `bd_c`
- `dbegin`
- `dend`

该组字段必须用于支撑：

- 现金科目识别
- 银行科目识别
- 银行对账启用口径
- 对账方向与期间控制

#### 6.2.5 控制与状态

首批核心字段包括：

- `bend`
- `bclose`
- `cgather`
- `bexchange`
- `itrans`
- `cother`
- `iotherused`

该组字段必须用于支撑：

- 末级控制
- 封存控制
- 汇总打印控制
- 汇兑损益参与控制
- 特殊科目标识
- 受控系统标识
- 其他系统使用状态

#### 6.2.6 扩展定义

首批核心字段包括：

- `company_id`
- `bcDefine1` 至 `bcDefine16`

该组字段必须用于支撑：

- 公司隔离
- 扩展定义能力
- 后续差异化账套属性扩展

### 6.3 首版实施规则

会计科目首版必须满足以下要求：

- 必须按公司隔离
- 必须支持层级与末级判断
- 必须支持启用 / 封存或等价状态控制
- 必须支持辅助核算标识维护
- 必须支持编码、名称、助记码查询
- 必须支持与凭证编辑器一致的可选项输出

会计科目首版不强制要求：

- 把 Excel 中每一个字段都开放到页面表单
- 一次性实现所有批量能力
- 一次性实现全量导入导出
- 一次性实现完整 U8 全字段编辑体验

但规范中必须明确：

- 哪些字段属于首批必做
- 哪些字段属于保留扩展
- 哪些字段属于受控变更字段

### 6.4 与账套模板和账套生成的关系

- 账套模板负责提供初始科目体系
- 账套创建负责生成公司级会计科目
- 会计档案模块负责账套创建后的科目查看、维护、停用和扩展属性管理
- 会计科目后续真实化时，不得绕过账套生成链路另造一套与账套脱节的科目源

### 6.5 禁止性规则

后续实现会计科目时，默认不得采用以下方式：

- 继续长期使用硬编码种子作为正式数据源
- 不按公司隔离直接读全表
- 把科目维护混入系统设置页面
- 把辅助核算标识做成无约束自由文本
- 在未定义审计与影响范围前直接支持高风险字段批量改写

## 7. 权限与审计规范

### 7.1 权限语义

会计档案模块后续至少应规范以下权限能力：

- 查看
- 创建
- 编辑
- 停用 / 删除
- 导入
- 导出

会计科目与项目档案除上述通用语义外，还应预留以下受控能力语义：

- 启用 / 封存
- 层级调整
- 辅助核算属性调整
- 受控字段变更

本次只定义权限语义边界，不在文档中提前写死具体权限编码。

### 7.2 审计要求

以下动作默认视为高风险动作，必须留痕：

- 会计科目新增
- 会计科目编辑
- 会计科目停用 / 封存
- 会计科目辅助核算属性调整
- 导入
- 导出
- 批量启停
- 批量调整

审计信息至少应记录：

- 操作者
- 操作时间
- 归属公司
- 目标对象
- 输入参数摘要
- 操作结果
- 失败原因

## 8. 与现有系统关系

### 8.1 与系统设置的关系

- 员工、部门当前若存在对系统设置数据源的复用，只能视为过渡实现
- 后续真实化时，产品定义必须回到财务域规范
- 不得因为底层复用系统设置表，就把会计档案直接视为系统设置子功能

### 8.2 与财务系统管理的关系

- 财务系统管理负责账套生命周期和模板初始化
- 会计档案负责账套生成后的公司级财务主数据维护
- 会计档案不得承担账套创建、账套备份、账套删除职责

### 8.3 与电子档案的关系

- 会计档案不复用 `archives:*` 的产品定义
- 不默认迁移或并入电子档案入口
- 名称相近不代表职责相同

## 9. 后续实施约束

后续任何会计档案相关页面、接口、权限、SQL、测试方案，都必须先对照本规范。

未在本规范中明确的高风险行为，默认不得直接实现。

统一实施优先级口径如下：

- 先出规范
- 再按规范拆页面、接口、数据结构、权限和测试
- 不允许先写功能后补规范

后续建议按以下优先级推进真实化：

- 第一优先级：会计科目
- 第二优先级：项目档案
- 第三优先级：部门档案
- 第四优先级：员工档案从查询型过渡到财务域受控档案
- 客户 / 供应商在现有真实页基础上做收敛式增强，不重做产品定义

### 9.1 前端显示与编码稳定性规范

- 页面展示不得依赖硬编码脏数据兜底；若后端字段缺失、异常或辅助核算数据暂不可用，页面必须局部降级并给出可读中文提示，不得直接展示乱码、对象串化结果或技术字段
- 涉及客户、供应商、员工、部门、项目、会计科目的列表、详情、选择器、辅助核算引用、导入导出结果、文件名、下载记录时，必须验证中文显示一致性

### 9.2 后端实现与运行稳定性规范

后续凡涉及会计档案后端接口、服务、导入导出、批量操作、辅助核算联动的新增或修改，必须统一按以下规则处理：

- 对会计科目、项目档案这类受控对象，涉及层级、辅助核算属性、启停状态、引用关系的改动必须具备幂等和失败收口意识，避免脏数据残留

## 12. 项目档案首版真实化补充口径（2026-04-08）

### 12.1 双对象模型
- 项目档案自本次真实化起，统一按“项目分类 + 项目主目录”双对象实施，不再把项目分类隐含为项目字段的附属概念。
- 项目分类负责承载公司级项目分类编码、分类名称、启停状态与分类维度排序；项目主目录负责承载公司级项目编码、项目名称、所属分类、启停状态、封存状态和结束日期等主数据。
- 后续任何项目档案相关页面、接口、权限、SQL、测试和总账联动设计，必须同时考虑这两个对象，不得只做项目主目录而忽略项目分类正式来源。

### 12.2 数据字段与主键口径
- 项目主目录主键统一使用通用 `id` 承接 Excel 中 `I_id` 的业务语义，不再重复建设并行的 `I_id` 字段。
- 项目主目录首版必须至少承接以下核心字段：`citemcode`、`citemname`、`bclose`、`citemccode`、`iotherused`、`dEndDate`、`company_id`。
- 为满足会计档案统一规范，项目主目录必须补充通用治理字段：`status`、`sort_order`、`created_by`、`updated_by`、`created_at`、`updated_at`。
- 项目分类至少必须具备：`company_id`、`project_class_code`、`project_class_name`、`status`、`sort_order`、审计字段。

### 12.3 状态控制口径
- 项目档案必须同时采用 `status` 与 `bclose` 双状态模型，不得把“停用”和“封存”混成一个字段语义。
- `status` 统一表示启用 / 停用，用于控制项目是否可作为当前业务正式可选对象。
- `bclose` 统一表示封存 / 未封存，用于控制项目是否进入后续新增凭证等运行态可选范围。
- 后续涉及项目启停、封存、解封、引用校验时，必须分别按两套状态语义处理，不得相互替代。

### 12.4 与总账联动口径
- 项目档案自本次真实化起，成为总账凭证中“项目分类 / 项目”辅助核算的正式主数据来源。
- 总账凭证元数据不得继续以硬编码项目分类种子或项目种子作为正式来源；后续取数必须来自公司级项目分类与项目主目录。
- 凭证保存与修改时，必须校验 `citemClass` 与 `citemId` 的归属关系、公司归属、启停状态与封存状态，避免出现项目分类与项目错配。
- 已停用或已封存项目不得进入新凭证默认可选项，但历史引用展示必须保持可追踪、可回看。

### 12.5 实施约束
- 项目档案后续若新增导入、导出、批量启停、批量封存、批量分类调整等重操作，必须先补规范再开发，不得绕开本规范直接落地。
- 若后续发现项目档案规范字段语义与新的字段定义来源发生直接冲突，必须先由需求提出人与产品口径裁决后再开发，工程实现不得自行拍板。
