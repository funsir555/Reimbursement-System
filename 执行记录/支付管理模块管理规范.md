# 支付管理模块管理规范

更新日期：2026-04-08

## 1. 文档定位

本文档是报销系统“支付模块”的正式管理规范与实施基线，不是原型说明、接口草稿、表结构草稿，也不是操作手册。
后续凡涉及支付模块相关功能的需求分析、设计评审、开发、联调、测试、验收与上线，均必须先对照本规范执行。

本文档主要规范以下支付相关能力：
- 付款单工作台
- 支付任务状态流转
- 银企直连配置
- 支付模块相关权限、SQL、测试与发布

本文档只规范支付模块自身，不展开以下模块的完整功能细则：
- 新建报销、我的报销、待我审批、单据查询等报销主链路
- 流程管理配置中心
- 总账、凭证生成、会计档案
- 首页工作台、下载中心、消息通知等公共能力

上述模块虽与支付模块存在上下游关系，但不属于本文档的主体实施范围；后续若要重定义边界，必须先修订规范，再实施代码。

自本文档发布之日起，其作为支付模块后续功能落地与修改的唯一实施基线。

## 2. 模块定义

支付模块是报销系统运行态业务中的支付执行子域，负责承接已进入支付链路的单据，并围绕付款执行、支付状态、回单结果和银企直连配置提供统一能力。

支付模块必须服务于以下业务场景：
- 支付执行人查看本人可处理的付款单
- 发起支付、手动标记已支付、标记支付异常
- 跟踪支付状态与回单状态
- 维护公司账户对应的银企直连配置
- 通过统一单据详情页查看付款单对应单据上下文

支付模块当前不负责以下内容：
- 流程模板、审批流、表单设计等配置能力
- 报销新建、重提、审批处理等主业务入口
- 总账凭证生成、会计档案维护
- 独立于报销系统之外的第二套支付产品入口

## 3. 核心口径与边界

### 3.1 模块归属口径

支付模块属于现有报销系统架构内的运行态支付子域，不是独立产品，不允许脱离现有报销管理体系平行再造第二套支付中心。
即：
- 菜单入口继续挂在左侧导航的“支付”父菜单下
- 页面风格继续沿用现有报销管理工作台布局、卡片、列表与详情联动体系
- 不允许把付款单、银企直连拆成与报销系统并列的独立站点或独立产品

### 3.2 前端路由口径

当前支付模块前端基线以以下路由为准：
- `/expense/payment/orders`
- `/expense/payment/bank-link`

付款单查看详情时，继续统一进入共享单据详情页：
- `/expense/documents/:documentCode`

后续新增支付模块前端能力，必须优先在上述既有路由域内演进，不得为相同业务语义再造第二套路由或第二套支付详情页。

### 3.3 后端接口域口径

支付模块后端接口继续统一收敛在：
- `/auth/expense-payment/**`

当前真实接口边界至少包括：
- `GET /auth/expense-payment/orders`
- `POST /auth/expense-payment/tasks/{taskId}/start`
- `POST /auth/expense-payment/tasks/{taskId}/complete`
- `POST /auth/expense-payment/tasks/{taskId}/exception`
- `GET /auth/expense-payment/bank-links`
- `GET /auth/expense-payment/bank-links/{companyBankAccountId}`
- `PUT /auth/expense-payment/bank-links/{companyBankAccountId}`
- `POST /auth/expense-payment/banks/cmb-cloud/callback`

后续新增支付能力必须优先在上述接口域下扩展，不得围绕付款单或银企直连另造平行 controller 域或新的 API 前缀。

### 3.4 权限口径

支付模块权限继续沿用以下命名空间：
- `expense:payment:payment_order:view`
- `expense:payment:payment_order:execute`
- `expense:payment:bank_link:view`
- `expense:payment:bank_link:edit`

后续不得将相同语义的支付能力散落到无关权限域，也不得绕开权限口径直接在前端放出入口。

### 3.5 当前现状基线

当前支付模块的真实实现基线按以下路径理解：
- 菜单入口：`frontend/admin-web/src/layouts/MainLayout.vue`
- 路由入口：`frontend/admin-web/src/router/index.ts`
- 付款单页面：`frontend/admin-web/src/views/expense/ExpensePaymentOrdersView.vue`
- 银企直连页面：`frontend/admin-web/src/views/expense/ExpenseBankLinkView.vue`
- 前端 API：`frontend/admin-web/src/api/index.ts` 中的 `expensePaymentApi`
- 后端 controller：`backend/auth-service/src/main/java/com/finex/auth/controller/ExpensePaymentController.java`
- 后端核心聚合服务：`backend/auth-service/src/main/java/com/finex/auth/service/impl/ExpenseDocumentServiceImpl.java`

当前付款单页面已经形成工作台式结构，并且页面内真实存在 5 个状态视图：
- 待支付 `PENDING_PAYMENT`
- 支付中 `PAYING`
- 已支付 `PAYMENT_COMPLETED`
- 已完成 `PAYMENT_FINISHED`
- 支付异常 `PAYMENT_EXCEPTION`

后续如果要将上述状态卡、状态码、页面分组或状态口径合并、删减、重命名，必须先判断是否与本规范冲突；若冲突，必须先询问你如何处理，再决定是修订规范还是调整实现。

当前路由中 `/expense/payment/bank-link` 的 `meta.title / tabTitle` 存在乱码，属于已知不合规示例；后续凡触达该路由配置，必须优先修正为 UTF-8 中文可读文案，并补对应测试。

## 4. 核心对象定义

### 4.1 付款单

付款单是支付模块的核心工作台对象，用于承接进入支付节点后的单据支付处理任务。
付款单至少应具备以下语义：
- `taskId`
- `documentCode`
- `documentTitle`
- `submitterName`
- `templateName` 或单据类型信息
- `paymentStatusCode / paymentStatusLabel`
- `receiptStatus`
- `amount`
- `paymentCompanyName`
- `taskCreatedAt`
- `paidAt`

付款单是支付执行视角的运行态对象，不是新的业务主单据类型；其本质上仍然隶属于报销系统中的单据实例。

### 4.2 支付任务

支付任务是流程命中支付节点后生成的待办执行对象，用于表示某位支付执行人当前是否可以处理某笔付款。
支付任务必须具备以下语义：
- 与单据实例一一关联到具体支付节点
- 具备执行人范围与可见范围
- 可区分未开始、处理中、异常、完成等状态
- 可被审计与追踪，不允许静默修改

### 4.3 支付状态

支付状态必须是标准化、可枚举、可追踪的正式状态，不得以自由文本替代。
当前真实实现至少包含以下状态：
- `PENDING_PAYMENT`：待支付
- `PAYING`：支付中
- `PAYMENT_COMPLETED`：已支付
- `PAYMENT_FINISHED`：已完成
- `PAYMENT_EXCEPTION`：支付异常

后续若需要扩展、合并或重命名支付状态，必须同步：
- 前端状态卡与状态标签
- 列表筛选与统计口径
- 详情页状态展示
- 后端 DTO / VO / 枚举与服务校验
- 自动化测试与人工回归用例

### 4.4 回单状态

回单状态用于描述银行回单查询结果，属于支付任务的附属状态，不得与支付状态混用。
当前至少应维持以下语义：
- `PENDING`：待回单
- `RECEIVED`：已收回单
- `FAILED`：回单失败

回单状态变更只应描述银行回执结果，不得被用作替代单据审批状态或付款任务主状态。

### 4.5 银企直连配置

银企直连配置是公司账户维度的支付通道配置对象，用于维护银行直连所需的提供方、通道、认证与回调能力。
当前真实实现至少包含以下语义：
- 公司信息与公司账户信息
- 银行提供方与直连通道
- operatorKey / callbackSecret / publicKeyRef
- 客户号、应用 ID、接口地址、协议版本
- 证书引用、密钥引用、签名算法、加密算法
- 是否启用直连
- 是否启用自动回单查询

银企直连配置属于公司账户主档的扩展配置，不是独立于公司账户之外的第二套账户体系。

### 4.6 公司账户与银行回调

公司账户是银企直连配置的承载对象，银行回调是支付状态自动推进的重要输入。
后续无论是手动支付还是银行回调，都必须围绕“付款单 -> 支付任务 -> 单据状态 -> 银行回单记录”这一链路实施，不得跳过已有链路直接改表兜底。

## 5. 统一业务规则

### 5.1 付款单与单据状态统一规则

- 付款单是单据进入支付链路后的支付执行视图，不是新的主业务单据。
- 付款单状态必须与单据状态保持统一语义，不允许页面显示一套、后端保存另一套。
- 付款完成后如何进入 `PAYMENT_COMPLETED` 或 `PAYMENT_FINISHED`，必须由统一服务层推进，不允许页面自行猜测终态。
- 后续若调整付款完成后的终态口径，必须先同步修订规范并评估对报销列表、详情页、凭证链路的影响。

### 5.2 支付执行人可见范围规则

- 付款单列表只应返回当前登录支付执行人本人可见的任务。
- 不允许把付款单做成“全员共享付款池”而不经明确权限设计。
- 执行人解析、支付节点权限与可见范围必须继续由后端控制，前端只做展示与交互。

### 5.3 支付动作统一规则

- `发起支付`、`手动已支付`、`标记异常` 必须属于受控动作，不允许通过前端拼装状态直改。
- 非法状态下不得开放不合法动作，例如已完成任务不允许再次发起支付。
- 支付异常是否允许重试，必须服从流程节点配置与服务层判断，不得由页面单独决定。
- 所有支付动作都必须保留日志、状态迁移与必要审计信息。

### 5.4 银企直连统一规则

- 银企直连配置继续挂靠公司账户，不另造独立账户主档。
- 同一公司账户的启用、停用、保存必须由统一保存逻辑处理，不允许各页面分别写入不同字段口径。
- 银企直连回调必须走标准接口与标准失败返回，不允许通过数据库脚本或后台手工改值代替正式链路。
- 首期银行通道若只支持招商银行云直连，应在页面和规范中明确，不得伪装成已支持全量银行。

### 5.5 详情与导航统一规则

- 付款单中的“查看”继续统一跳转共享单据详情页，不允许额外新造支付专属详情页。
- 详情页中的支付信息、回单信息、时间线必须继续挂在统一详情模型下展示。
- 返回链路必须优先回到来源语义对应的支付页面，不得出现串页、错路由或落回无关模块。

### 5.6 支付管理防跑偏清单

后续每次修改支付模块前，至少先逐项确认：
- 是否仍在既有支付菜单、既有路由和 `/auth/expense-payment/**` 接口域内演进。
- 是否仍复用统一单据详情页、统一支付状态口径和既有 `expensePaymentApi`。
- 是否所有用户可见中文、状态文案、按钮标题、路由标题都经过 UTF-8 与实际显示检查。
- 是否支付执行、异常标记、银企直连配置、银行回调都补了对应回归测试。
- 是否评估了对报销主链路、付款状态、凭证生成、通知下载等上下游模块的影响。
- 如果本次需求与本规范冲突，是否已经先询问你如何处理；未询问前不得直接开发或擅自拍板修改规范。

## 6. 前端实施规范

### 6.1 菜单与路由规范

- 支付菜单继续挂在 `MainLayout.vue` 的“支付”父菜单下，不得挪到报销管理之外的无关父级。
- `frontend/admin-web/src/router/index.ts` 是支付模块路由元信息的唯一前端基线。
- 付款单继续由 `ExpensePaymentOrdersView.vue` 承载，银企直连继续由 `ExpenseBankLinkView.vue` 承载。
- 不允许为付款单、支付状态、回单查询再造第二套并行页面入口。

### 6.2 工作台与交互规范

- 付款单页面继续保持工作台式布局：状态卡 + 工具栏 + 列表区。
- 当前页面内已经形成 `tab` 驱动的多状态视图；后续若变更状态卡数量、文案、顺序或 query 结构，必须先对照本规范评估。
- 列表操作按钮必须与后端真实能力对齐，不允许前端出现不存在的动作入口。
- 付款单页面的详情跳转继续走共享详情页，不得复制出新的支付详情组件。

### 6.3 文案与编码规范

- 所有用户可见中文、状态标签、按钮标题、路由 `meta.title / tabTitle / description` 必须以 UTF-8 保存，并确保页面实际可正常显示。
- 禁止提交乱码字符串、错误转码后的中文、英文占位标题长期留存。
- 当前 `/expense/payment/bank-link` 路由元信息存在乱码，属于不合规示例；后续凡触达该配置，必须顺手修正并补测试。

### 6.4 组件与样式复用规范

- 支付模块必须优先复用现有报销工作台样式体系、表格样式、状态标签、详情跳转模式与公共 helper。
- 不允许无必要新增平行的支付状态体系、平行表格组件或第二套侧边工作台风格。
- 银企直连页面继续复用现有卡片、表单、按钮和空态风格，不另造完全不同的视觉语义。

### 6.5 前端测试规范

后续触达支付模块前端能力时，至少应按变更范围补齐或回归以下测试基线：
- `frontend/admin-web/src/views/expense/__tests__/ExpensePaymentOrdersView.test.ts`
- `frontend/admin-web/src/views/expense/__tests__/ExpenseBankLinkView.test.ts`
- `frontend/admin-web/src/layouts/__tests__/MainLayout.test.ts`
- `frontend/admin-web/src/router/__tests__/index.test.ts`

若变更涉及状态卡、详情跳转、路由标题、权限显隐或银企直连保存逻辑，必须补对应断言，避免再次出现乱码、错菜单、错路由或动作错显。

## 7. 后端实施规范

### 7.1 分层规范

- `ExpensePaymentController` 只负责权限校验、入参绑定、结果包装与 HTTP 边界。
- 支付模块的核心聚合与状态推进继续由 `ExpenseDocumentService` / `ExpenseDocumentServiceImpl` 负责。
- 不允许在 controller 中直接拼装复杂支付逻辑、直接写 mapper，或临时绕开服务层完成状态迁移。
- 银企直连、银行回调、付款单列表、支付动作等能力必须继续收敛在统一服务域中处理。

### 7.2 接口与异常规范

- 所有支付接口必须继续返回标准结果结构，不能出现半结构化响应。
- 不允许吞异常、无响应、长时间 pending，或依赖前端超时来兜底后端错误。
- 发起支付、标记已支付、标记异常、保存银企直连配置等高风险动作，必须给出稳定且可理解的失败结果。
- 银行回调接口必须可审计、可追踪、可重放验证，不允许因异常回调导致系统静默错乱。

### 7.3 DTO / VO 与状态同步规范

- 支付相关 DTO / VO、状态标签、权限码或页面字段变更时，必须同步更新前端类型、后端测试和本文档。
- `ExpensePaymentOrderVO`、银企直连配置 VO / DTO、支付状态标签与回单状态标签都属于强同步对象，不允许只改单侧。
- 任何状态迁移规则变化都必须同步评估 `ExpenseDocumentDetailVO` 与统一详情展示。

### 7.4 状态迁移与动作约束规范

- 支付任务状态迁移必须由服务层统一控制，前端只触发动作，不直接写状态值。
- 非法状态下必须拒绝执行不允许的支付动作。
- 手动已支付与银行回调已支付都必须走统一的完成逻辑，避免出现两套完成语义。
- 标记异常与允许重试等逻辑必须绑定支付节点配置与任务上下文，不得靠页面自由组合。

### 7.5 审计与日志规范

- 所有支付动作必须保留可追踪日志，至少包括动作人、动作时间、节点、备注和状态迁移结果。
- 银企直连配置保存、启停、回调处理都必须具备审计语义，不能只改配置不留痕。
- 若后续新增自动轮询、自动回单、自动重试等后台能力，必须同步补充日志与异常处理约束。

## 8. 数据与 SQL 规范

- 支付模块相关数据变更只允许走增量 SQL，不允许依赖整库初始化兜底。
- 字符集统一使用 UTF-8 / utf8mb4，禁止把乱码数据写入数据库后再由前端硬修文案。
- 涉及支付模块的数据调整，应优先围绕现有单据实例、任务、银行支付记录、公司账户与权限表扩展，不得轻易再造平行主表。
- 任何结构变更、字典扩展、权限脚本、种子数据都必须具备可重复执行和现网增量升级能力。
- 禁止通过手工直改数据库绕过正式支付状态迁移链路来“修好页面”。

## 9. 权限与审计规范

- 支付模块权限继续锁定在 `expense:payment:*` 命名空间，不得发散到无关权限树。
- `view` 与 `execute` 必须分离，查看付款单和执行支付动作不可混为一谈。
- 银企直连的查看与编辑权限也必须分离，不允许默认所有查看者均可修改配置。
- 支付执行、异常标记、配置保存、银行回调处理都属于高风险动作，必须保留可审计记录。

## 11. 与现有系统关系

### 11.1 与报销管理模块关系

支付模块是报销运行态主链路的下游执行域，消费的是单据、详情和状态体系，不得反向重定义报销主链路页面。
付款单查看详情时，必须继续复用报销管理的统一详情页。

### 11.2 与流程管理模块关系

支付模块依赖流程管理中支付节点的配置结果，但不负责定义流程模板、支付节点设计器或审批规则本身。
如需改动支付执行人解析、允许重试等节点语义，必须同步评估流程管理模块规范。

### 11.3 与凭证生成 / 总账关系

支付模块可影响单据的业务状态与支付完成时点，但不负责直接重定义凭证生成和总账入账规则。
如支付状态变化会影响凭证链路，必须同步评估相关模块，而不是在支付模块内私自扩展总账逻辑。

### 11.4 与下载中心 / 消息通知关系

支付模块如需导出、通知或异步任务提醒，应继续复用现有下载中心与通知中心，不得在支付模块内部平行再造第二套下载或通知产品。
