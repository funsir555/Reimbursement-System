## 1. 文档定位与适用范围

本规范是当前 `finex_db` 的统一数据库管理基线，后续所有数据库相关操作都必须以本规范为准执行。这里的数据库操作包括但不限于：建表、加列、改列、补数、修复数据、增加索引、变更外键、修改注释、迁移历史库、回滚、删除数据、停用数据、初始化新库、修复乱码和验收检查。

本规范的目标不是说明产品功能，而是约束数据库管理过程，重点防止以下四类问题再次发生：

1. 误删数据，导致业务数据无法恢复。
2. 前端出现乱码，尤其是中文名称、注释、下拉选项、页面标题、导入导出内容乱码。
3. 后端因为字段缺失、类型不匹配、结构漂移、索引缺失或脚本执行顺序错误而报错。
4. 数据表和字段没有中文注释，导致后续维护、排查和交接成本过高。

本规范默认适用于当前仓库 `C:\Users\funsir\Desktop\报销系统` 对应的数据库管理工作，并优先于各模块自己的零散说明。若后续模块规范与本规范冲突，以本规范为准。

关于连接口径：当前实际可连接且已核实的数据库实例是 `127.0.0.1:3306/finex_db`。因此本规范以 `3306` 为主执行口径。

## 3. 统一管理原则

后续所有数据库操作必须遵守以下硬性原则，不允许留实现弹性：

### 3.1 先脚本，后执行

任何数据库变更，必须先落地成 SQL 脚本，再执行。禁止直接在客户端手工点改结构、改数据、改注释后不留脚本。

### 3.2 新库走初始化，旧库走迁移

- 新库只执行 `backend/sql/init*.sql`
- 旧库升级只执行 `backend/sql/migrate*.sql`
- 权限刷新只执行 `backend/sql/refresh*.sql`
- 禁止把初始化脚本直接当增量修复脚本乱跑
- 禁止把迁移脚本当新库首发脚本混用

### 3.3 先评估，后操作

所有执行都必须先完成以下动作：

1. 核对当前结构。
2. 确认影响范围。
3. 准备备份或回滚方案。
4. 登记执行记录。
5. 执行后做验收。

### 3.4 生产性删除默认为高风险

任何删除操作默认视为高风险动作。能用停用、状态位、软删除解决的，不允许直接物理删除。禁止先删数据再想办法恢复。

### 3.5 数据库变更必须与前后端联动验收

数据库变更不只看 SQL 是否执行成功，还必须确认：

- 前端页面不乱码。
- 接口返回字段不丢失。
- 后端实体、Mapper、DTO、SQL 映射不报错。
- 关键业务流程能正常跑通。

## 4. 变更分级与审批要求

为了防误删、防误改，数据库变更按风险分为四级：

### 4.1 一级：低风险结构补充

适用范围：仅增加注释、补表注释、补字段注释、增加不影响现有逻辑的索引、增加可空字段。

要求：

- 必须有脚本。
- 必须先核对现有字段定义。
- 必须执行后校验 `information_schema`。
- 可以在低峰时段执行，但仍需留执行记录。

### 4.2 二级：中风险结构变更

适用范围：修改列类型、长度、默认值、排序规则，增加外键、唯一约束，调整索引策略。

要求：

- 必须先导出 `SHOW CREATE TABLE`。
- 必须确认是否影响历史数据、查询性能、后端映射、前端页面。
- 必须有回滚脚本或回滚步骤。
- 必须在执行前后都做结构对比。

### 4.3 三级：高风险数据修复

适用范围：批量补数、批量修状态、批量纠正编码、批量改主数据、历史数据修复。

要求：

- 执行前必须先做 `SELECT` 预览和备份。
- 必须记录预计影响行数。
- 必须限制明确 `WHERE` 条件。
- 必须有恢复原数据的方案。

### 4.4 四级：极高风险删除或不可逆操作

适用范围：物理删除数据、删表、删列、删索引后可能影响运行、`TRUNCATE`、`DROP`、重建主键或大规模重构。

要求：

- 原则上禁止直接执行。
- 确需执行时，必须先完成：审批记录、完整备份、恢复脚本、执行窗口、停机/影响说明、执行后专项验收。
- 未满足前置条件，一律不得执行。

## 5. DDL 规范

DDL 指建表、改表、增列、改列、删列、索引、约束、注释、字符集、排序规则等结构类脚本。

### 5.1 所有新表必须满足的最低要求

所有新表必须同时满足以下要求：

- 显式写 `ENGINE=InnoDB`
- 显式写 `DEFAULT CHARSET=utf8mb4`
- 必须有中文表注释 `COMMENT='中文表名'`
- 所有业务字段必须有中文字段注释
- 所有金额字段必须使用 `DECIMAL(18,2)`，不得使用 `FLOAT`、`DOUBLE` 或不定精度方案存储货币值
- 时间字段命名优先统一为 `created_at`、`updated_at`
- 状态字段必须写清枚举语义，例如 `状态:1启用 0停用`
- 布尔字段必须写清 `1是/0否`

### 5.2 新增字段规则

新增字段时必须：

- 明确字段用途、类型、长度、是否可空、默认值
- 金额字段统一使用 `DECIMAL(18,2)`，并在注释中写清业务含义
- 费率、汇率、数量、工作量等非金额字段不得因名称相似被误设为金额类型
- 明确是否需要索引、唯一约束、外键
- 补齐中文注释
- 先确认实体、DTO、Mapper、前端是否需要同步变更

### 5.3 修改字段规则

涉及已有表改列时，必须先执行：

- `SHOW CREATE TABLE <表名>;`
- 查询 `information_schema.COLUMNS`
- 查询 `information_schema.STATISTICS`
- 查询 `information_schema.KEY_COLUMN_USAGE`

然后生成精确的 `ALTER TABLE ... MODIFY COLUMN ...` 语句。修改字段注释时，必须保持以下属性完全不变：

- 原类型
- 原长度
- `NULL / NOT NULL`
- 默认值
- `AUTO_INCREMENT`
- `ON UPDATE`
- 索引
- 唯一约束
- 外键

禁止只为了改注释，就顺手改字段类型、改默认值、改可空性。

### 5.4 索引与外键规则

- 外键、唯一约束、索引变更必须单独说明影响面。
- 禁止在无性能评估和无回滚方案的情况下删除现有索引。
- 禁止在不知道业务引用关系时删除外键。
- 任何外键新增前，必须先确认历史数据满足引用完整性。

### 5.5 排序规则与字符集规则

- 新表、新字段默认按 `utf8mb4` 口径设计。
- 不允许使用 `latin1`、`gbk`、`utf8`（3字节）等历史口径。
- 若需调整现有表排序规则，必须单独评估，不允许和普通业务变更混在一起执行。

## 6. DML 规范

DML 指插入、更新、删除、补数、清洗数据、初始化种子数据等数据类操作。

### 6.1 更新和删除前必须先预览

任何 `UPDATE` / `DELETE` 执行前，必须先写对应 `SELECT` 预览 SQL，并在执行记录中登记：

- 预览 SQL
- 预计影响行数
- 筛选条件说明
- 风险说明

没有预览 SQL，不允许执行正式更新或删除。

### 6.2 禁止无条件批量更新或删除

以下写法原则上禁止：

- `UPDATE 表名 SET ...;`
- `DELETE FROM 表名;`
- `TRUNCATE TABLE 表名;`

所有批量改数必须带明确 `WHERE` 条件，且条件必须能通过 `SELECT` 复核出相同影响范围。

### 6.3 幂等脚本优先

凡是初始化、补数、基础数据修复脚本，优先写成幂等方式，例如：

- `INSERT ... SELECT ... WHERE NOT EXISTS`
- `INSERT ... ON DUPLICATE KEY UPDATE`
- 先判断对象存在再 `ALTER TABLE`

禁止写“一次能跑，第二次报错”的脆弱脚本作为长期维护脚本。

### 6.4 历史数据修复必须先备份

涉及以下场景时，必须先对目标数据集做备份表或导出：

- 状态修复
- 历史补数
- 编码修复
- 批量替换值
- 主数据修正

备份方式至少满足以下之一：

- 备份到独立 SQL 文件
- 备份到 CSV / Excel 导出文件
- 备份到带日期后缀的临时表

## 7. 删除与回滚规范

### 7.1 默认禁止事项

以下操作默认禁止：

- `DROP DATABASE`
- 无备份 `TRUNCATE`
- 无条件 `DELETE`
- 不可恢复的批量物理删除
- 无回滚方案的删列
- 无评估的删索引、删外键

### 7.2 物理删除前置条件

确需物理删除时，必须先完成以下条件：

1. 确认影响范围。
2. 确认无上游/下游依赖或已完成联动处理。
3. 完成备份快照。
4. 编写恢复脚本。
5. 写清审批记录。
6. 写清执行窗口。
7. 写清执行后验证方法。

只要有一项缺失，就不得执行物理删除。

### 7.3 回滚要求

任何结构变更都必须同步给出回滚方案，至少包含：

- 如何恢复旧字段定义
- 如何恢复旧索引
- 如何恢复旧外键
- 如何恢复旧默认值
- 如何恢复旧注释
- 如何恢复被修改前的数据（如涉及 DML）

如果无法回滚，必须在执行记录中明确写成“不可逆变更”，并单独标红提示。

## 8. 编码、字符集与注释规范

这是本规范的重点章节，直接关系到前端乱码、后端 bug 和数据库可维护性。

### 8.1 SQL 文件编码统一要求

- SQL 文件统一使用 UTF-8 保存。
- 禁止使用 GBK、ANSI、UTF-8 with BOM 混乱口径、混合编码文件。
- SQL 文件头统一写 `SET NAMES utf8mb4;`
- 含中文注释的 SQL 文件，提交前必须做字节级确认，确保文件里是真实中文，而不是 `?`。

### 8.2 连接串统一要求

JDBC / 应用连接串统一使用 UTF-8 口径，例如当前项目使用：

`jdbc:mysql://localhost:3306/finex_db?useUnicode=true&characterEncoding=utf-8&useSSL=false&serverTimezone=Asia/Shanghai`

后续若改连接串，不得删除 `useUnicode=true` 与 `characterEncoding=utf-8` 相关设置。

### 8.3 表注释与字段注释要求

- 所有新表必须有中文表注释。
- 所有新字段必须有中文字段注释。
- 所有金额字段必须有统一货币类型定义。
- 金额字段统一使用 `DECIMAL(18,2)`，禁止使用 `FLOAT`、`DOUBLE` 或不定精度方案存储货币值。
- 费率、汇率、数量、工作量、比例等非金额字段不适用金额字段规则。
- 所有历史整改表在修改时，应顺手补齐中文注释，但不得在同一变更中顺手改坏其它结构。
- 注释风格统一为“业务名 + 必要枚举说明”。

示例：

- `状态:1启用 0停用`
- `来源类型:MANUAL/DINGTALK/WECOM/FEISHU`
- `业务场景模式:PREPAY_UNBILLED预付未开票/INVOICE_FULL_PAYMENT到票全额付款`
- `默认路由:1是 0否`

### 8.4 注释禁止事项

新增或修改注释时，禁止出现以下内容：

- `column comment`
- `table comment`
- `comment here`
- `TODO`
- `?`
- `????`
- 纯英文占位词
- 与实际业务不一致的字面翻译

### 8.5 历史乱码反面案例

以下问题已在本轮整改中完成清理，后续写注释时必须作为历史反面案例，避免重复发生：

- `gl_customer` 当前大量字段注释为 `????`
- `fin_account_set*` 当前大量字段注释为 `????`
- `sys_user.stat_*` 当前字段注释为 `??????`
- `gl_vender.receipt_*` 当前字段注释为 `?????`

### 8.6 排查乱码的标准动作

执行注释或编码整改后，必须至少校验以下内容：

- SQL 文件保存编码是否为 UTF-8
- 表排序规则与字段字符集是否符合预期
- `information_schema.COLUMNS.COLUMN_COMMENT` 是否显示正常中文
- 前端页面、接口 JSON、导出文件是否正常显示中文

### 9.3 执行后固定验收清单

每次变更执行后，至少完成以下核查：

1. 表/列注释完整性核查。
2. 字符集/排序规则核查。
3. `?` / `????` 乱码注释核查。
4. 关键外键/索引存在性核查。
5. 后端启动后是否有 SQL 异常。
6. 前端页面是否出现中文乱码或接口字段缺失。
7. 关键业务流程是否能正常访问或提交。

### 9.4 推荐核查 SQL

```sql
-- 1. 查无表注释的表
SELECT TABLE_NAME
FROM information_schema.TABLES
WHERE TABLE_SCHEMA = 'finex_db'
  AND TABLE_TYPE = 'BASE TABLE'
  AND (TABLE_COMMENT IS NULL OR TABLE_COMMENT = '');

-- 2. 查无字段注释的列
SELECT TABLE_NAME, COLUMN_NAME
FROM information_schema.COLUMNS
WHERE TABLE_SCHEMA = 'finex_db'
  AND (COLUMN_COMMENT IS NULL OR COLUMN_COMMENT = '');

-- 3. 查乱码注释
SELECT TABLE_NAME, COLUMN_NAME, COLUMN_COMMENT
FROM information_schema.COLUMNS
WHERE TABLE_SCHEMA = 'finex_db'
  AND (COLUMN_COMMENT LIKE '%?%' OR COLUMN_COMMENT LIKE '%？？%');

-- 4. 查表排序规则
SELECT TABLE_NAME, TABLE_COLLATION
FROM information_schema.TABLES
WHERE TABLE_SCHEMA = 'finex_db'
  AND TABLE_TYPE = 'BASE TABLE'
ORDER BY TABLE_NAME;

-- 5. 查字段定义
SELECT TABLE_NAME, COLUMN_NAME, COLUMN_TYPE, IS_NULLABLE, COLUMN_DEFAULT, COLUMN_COMMENT
FROM information_schema.COLUMNS
WHERE TABLE_SCHEMA = 'finex_db'
  AND TABLE_NAME = '目标表名'
ORDER BY ORDINAL_POSITION;

-- 6. 查索引
SELECT TABLE_NAME, INDEX_NAME, COLUMN_NAME, NON_UNIQUE
FROM information_schema.STATISTICS
WHERE TABLE_SCHEMA = 'finex_db'
  AND TABLE_NAME = '目标表名'
ORDER BY INDEX_NAME, SEQ_IN_INDEX;
```

## 10. 禁止事项清单

以下事项一律禁止，除非另有单独审批并具备完整回滚与备份方案：

1. 禁止直接在客户端手工修改结构后不留 SQL 脚本。
2. 禁止把 `init*.sql` 直接当旧库增量脚本使用。
3. 禁止无条件 `UPDATE`、无条件 `DELETE`、无备份 `TRUNCATE`。
4. 禁止在不知道当前真实结构的情况下直接 `ALTER TABLE ... MODIFY COLUMN ...`。
5. 禁止把中文注释写成英文占位、`column comment`、`table comment`、`????`。
6. 禁止使用 GBK、ANSI 或未知编码保存 SQL 文件。
7. 禁止在未做评估的情况下批量统一所有表的字符集或排序规则。
8. 禁止删除外键、索引、字段后不验证前后端影响。
9. 禁止执行后不做 `information_schema` 验收核查。
10. 禁止数据库操作不留执行记录。

## 附：后续数据库操作的最低执行顺序

后续任何数据库变更，默认按以下顺序执行：

1. 明确目标与影响范围。
2. 核对现有表结构与字段定义。
3. 编写 SQL 脚本。
4. 准备备份与回滚方案。
5. 写执行记录草稿。
6. 先执行预览 SQL。
7. 正式执行 SQL。
8. 执行后校验结构、数据、注释、字符集。
9. 验证后端日志与前端页面。
10. 完成执行记录归档。

本规范发布后，后续所有数据库增删改操作，都必须先参照本规范执行；若实际需求必须突破本规范，必须先补充书面例外说明，再实施变更。
