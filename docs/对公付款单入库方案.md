# 对公付款单入库方案

## 1. 结论
- 张三提交对公付款单时报“系统异常”，高概率不是因为主表单里的业务控件没有一字段一列落库。
- 当前代码会在提交时同时写入审批单主表、费用明细表、审批任务表、审批日志表；如果运行时表或缺列未补齐，就会在提交阶段直接抛出 SQL 异常。
- 这次最可疑的根因是数据库仍停留在旧版结构，尤其是 `pm_document_expense_detail`、`pm_document_instance`、`pm_document_task` 的运行时字段没有补齐。

## 2. 当前入库设计
### 2.1 主单与明细是否分表
是，费用审批单和费用明细单是分表存储：
- 主表：`pm_document_instance`
- 明细表：`pm_document_expense_detail`
- 审批任务表：`pm_document_task`
- 审批动作日志表：`pm_document_action_log`

### 2.2 主单与费用明细如何关联
前端展示费用审批单与费用明细单，数据库主要通过以下字段关联：
- 主关联字段：`pm_document_expense_detail.document_code = pm_document_instance.document_code`
- 明细定位字段：`pm_document_expense_detail.detail_no`

对应到前端链路：
- `documentCode` 对应数据库 `document_code`，用于把一张审批单和它下面的多条费用明细关联起来。
- `detailNo` 对应数据库 `detail_no`，用于定位同一审批单下的一条具体费用明细。

### 2.3 动态表单字段如何落库
当前系统不是把每个表单控件都拆成数据库列，而是保留动态 JSON 入库：
- 主表单运行时数据：`pm_document_instance.form_data_json`
- 每条费用明细运行时数据：`pm_document_expense_detail.form_data_json`

因此像“付款公司、收款人、发票附件、说明”等动态控件，不建议为了这次异常单独拆列。当前满足需求的做法是：
- 运行时检索、统计、流程判断需要的字段保留独立列
- 其他动态字段继续保存在 JSON 中

## 3. 本次提交链路依赖的关键运行时字段
根据 `ExpenseDocumentServiceImpl` 当前实现，对公付款 / 企业往来相关费用明细在保存时会额外写入以下标准列：
- `expense_type_code`
- `business_scene_mode`
- `invoice_amount`
- `actual_payment_amount`
- `pending_write_off_amount`

同时审批运行时还依赖以下主表 / 任务表字段：
- `pm_document_instance.current_node_key`
- `pm_document_instance.current_node_name`
- `pm_document_instance.current_task_type`
- `pm_document_instance.finished_at`
- `pm_document_task.task_kind`
- `pm_document_task.source_task_id`

如果这些字段缺失，提交审批单时就可能出现“系统异常”。

## 4. 满足需求的入库方案（不删表）
### 4.1 首选方案
直接执行仓库里的幂等增量脚本：
- `backend/sql/init_expense_create_incremental.sql`

这个脚本会按“只增不删”的方式补齐费用审批创建链路所需的表、字段、索引，适合作为现场统一修复方案。

### 4.2 最小修复方案
如果现场不方便跑整包增量脚本，可以按最小范围执行：
- `backend/sql/migrate_expense_submit_runtime_fields.sql`

这个脚本同样是幂等的，专门补齐对公付款单提交审批最容易缺失的运行时结构：
- `pm_document_expense_detail`
  - `expense_type_code`
  - `business_scene_mode`
  - `invoice_amount`
  - `actual_payment_amount`
  - `pending_write_off_amount`
  - `idx_pm_document_expense_detail_document (document_code, sort_order)`
  - `idx_pm_document_expense_detail_design (detail_design_code)`
- `pm_document_instance`
  - `current_node_key`
  - `current_node_name`
  - `current_task_type`
  - `finished_at`
- `pm_document_task`
  - `task_kind`
  - `source_task_id`
  - `idx_pm_document_task_source (source_task_id)`

## 5. 推荐排查顺序
1. 检查 `pm_document_expense_detail` 是否存在。
2. 检查它是否缺少上述 5 个企业往来标准列。
3. 检查 `pm_document_instance` 是否缺少当前节点运行时列。
4. 检查 `pm_document_task` 是否缺少 `task_kind`、`source_task_id`。
5. 如果现场库是旧库，优先执行 `backend/sql/init_expense_create_incremental.sql`。
6. 补齐后再让张三重新提一张对公付款单，确认以下结果：
   - `pm_document_instance` 新增 1 条主单
   - `pm_document_expense_detail` 新增 N 条明细
   - 明细表 `document_code` 与主单一致
   - `pm_document_task` / `pm_document_action_log` 正常落库

## 6. 本次代码层配套
本次仓库已同步补充：
- 新增脚本 `backend/sql/migrate_expense_submit_runtime_fields.sql`
- `GlobalExceptionHandler` 额外识别 `current_node_key`、`current_node_name`、`current_task_type`、`finished_at` 等缺列异常，并统一提示先执行 `backend/sql/init_expense_create_incremental.sql`

## 7. 最终建议
- 不删表。
- 缺字段时按增量脚本补字段、补索引、补缺失表。
- 费用审批单与费用明细单继续保持主子表设计，通过 `document_code` 关联。
- 动态表单字段继续保留 JSON 入库，不建议因为这次异常改成一控件一列。
