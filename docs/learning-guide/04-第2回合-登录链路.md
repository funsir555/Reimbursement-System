# 第 2 回合：登录链路

这一回合只研究一个问题：

“为什么输入账号密码以后，系统能登录成功并进入后台？”

你学完后，不需要会写登录功能，但应该能说清：

- 前端从哪里发起登录
- 后端在哪里验证账号密码
- token 是谁生成的
- 登录后用户信息和权限是怎么补上的

---

## 本回合的主调用链

先背下来：

`/login 路由 -> LoginView.vue -> authApi.loginByPassword -> request(core.ts) -> /auth/login -> AuthController -> UserServiceImpl -> AuthLoginDomainSupport -> UserMapper / sys_user -> JwtUtil -> Result<LoginVO>`

登录成功之后，系统还有一条补充链路：

`router.beforeEach -> authApi.getCurrentUser -> /auth/me -> MvpController.me`

这条补充链路是为了拿当前用户资料和权限。

---

## 业务翻译版

你可以把登录过程理解成：

1. 用户进入登录页。
2. 页面收集用户名和密码。
3. 前端把用户名和密码发给后端。
4. 后端查 `sys_user` 表。
5. 后端比对密码、检查用户状态。
6. 后端生成 token。
7. 前端把 token 存进浏览器。
8. 后续访问页面时，前端带着 token 去取当前用户和权限。

## 文件卡片 1：登录入口路由

文件：

- `frontend/admin-web/src/router/route-catalog.ts`

它是干什么的：

- 声明 `/login` 这个页面路径对应哪个页面文件。

谁会调用它：

- 前端路由系统。

它再调用谁：

- `frontend/admin-web/src/views/LoginView.vue`

输入 / 输出是什么：

- 输入：浏览器访问 `/login`
- 输出：显示登录页面

业务规则是什么：

- 这是登录页入口，不是业务处理层。

风险点是什么：

- 如果这里路径写错，登录页根本进不去。

你只需要记住的一句话：

- “路由文件决定你从哪个地址能进到登录页。”

---

## 文件卡片 2：登录页面

文件：

- `frontend/admin-web/src/views/LoginView.vue`

它是干什么的：

- 展示登录表单，收集用户名和密码，点击按钮后触发登录请求。

谁会调用它：

- 路由系统。

它再调用谁：

- `frontend/admin-web/src/api/modules/auth.ts`

输入 / 输出是什么：

- 输入：用户名、密码
- 输出：调用登录接口；成功后把 token 和用户信息写入 `localStorage`

业务规则是什么：

- 页面层会做最基本的空值校验。
- 真正的账号密码校验不在页面里，在后端。

风险点是什么：

- 前端只做空值校验，不代表安全校验已经做完。
- 如果成功后没有正确保存 token，后续页面会被当成未登录。

你只需要记住的一句话：

- “LoginView 负责收集并发送登录信息，不负责最终判定对错。”

---

## 文件卡片 3：认证 API

文件：

- `frontend/admin-web/src/api/modules/auth.ts`

它是干什么的：

- 把登录页面的动作翻译成实际 HTTP 请求。

谁会调用它：

- `LoginView.vue`
- 路由守卫里的当前用户加载逻辑

它再调用谁：

- `frontend/admin-web/src/api/modules/core.ts`

输入 / 输出是什么：

- 输入：用户名、密码
- 输出：请求 `/auth/login`；另一个方法请求 `/auth/me`

业务规则是什么：

- 这里只定义请求地址和请求方式，不处理真正业务。

风险点是什么：

- 如果接口地址写错，页面会报登录失败。
- 同一个前端模块里的两个方法，可能对应不同后端 Controller。

你只需要记住的一句话：

- “auth.ts 负责告诉前端：登录该打哪个后端地址。”

---

## 文件卡片 4：前端通用请求层

文件：

- `frontend/admin-web/src/api/modules/core.ts`

它是干什么的：

- 统一处理前端对后端的请求。

谁会调用它：

- 各个 `api/modules/*.ts`

它再调用谁：

- 浏览器原生 `fetch`

输入 / 输出是什么：

- 输入：接口地址、请求方式、body、超时设置
- 输出：统一的接口结果对象

业务规则是什么：

- 自动带上 `Authorization: Bearer token`
- 接口 401 时清除登录态并跳回登录页
- 统一判断后端返回的 `code`

风险点是什么：

- 如果这里的 token 处理坏了，整个系统很多页面都会受影响。
- 如果统一错误处理写错，可能出现“明明后端返回错了，前端却没正确提示”。

你只需要记住的一句话：

- “core.ts 是前端所有请求共享的总管。”

---

## 文件卡片 5：后端登录入口

文件：

- `backend/auth-service/src/main/java/com/finex/auth/controller/AuthController.java`

它是干什么的：

- 接收 `/auth/login` 请求，把登录参数交给用户服务处理，再把结果包装返回。

谁会调用它：

- 前端的 `authApi.loginByPassword`

它再调用谁：

- `UserService`

输入 / 输出是什么：

- 输入：`LoginDTO`
- 输出：`Result<LoginVO>`

业务规则是什么：

- 接口入口接收用户名和密码。
- 出错时会把异常转换成更容易理解的错误信息。

风险点是什么：

- 如果这里只接请求、不做异常翻译，前端拿到的错误可能很难懂。
- 如果对外暴露过多底层错误细节，也可能带来安全或运维风险。

你只需要记住的一句话：

- “AuthController 是后端登录窗口，但真正校验密码的人不是它。”

---

## 文件卡片 6：登录参数 DTO

文件：

- `backend/auth-service/src/main/java/com/finex/auth/dto/LoginDTO.java`

它是干什么的：

- 定义登录接口收什么字段。

谁会调用它：

- `AuthController`

它再调用谁：

- 不直接调用别人，它是数据容器。

输入 / 输出是什么：

- 输入：`username`、`password`
- 输出：给后端后续逻辑使用

业务规则是什么：

- 用注解做非空校验。

风险点是什么：

- 如果参数校验缺失，后端可能要在更深层才发现空值问题。

你只需要记住的一句话：

- “LoginDTO 决定了登录接口最少要收什么字段。”

---

## 文件卡片 7：用户服务实现

文件：

- `backend/auth-service/src/main/java/com/finex/auth/service/impl/UserServiceImpl.java`

它是干什么的：

- 把登录和权限相关逻辑交给更细的领域支持类处理。

谁会调用它：

- `AuthController`

它再调用谁：

- `AuthLoginDomainSupport`
- `AuthAuthorizationDomainSupport`

输入 / 输出是什么：

- 输入：`LoginDTO`
- 输出：`LoginVO`

业务规则是什么：

- 这里更像业务中转和组织层。
- 真实的登录判断被拆到了 `AuthLoginDomainSupport`。

风险点是什么：

- 如果你只看 `UserServiceImpl`，会误以为它什么都没做；其实关键逻辑被拆出去了。

你只需要记住的一句话：

- “UserServiceImpl 是登录业务的中转站，不是最终细节所在。”

---

## 文件卡片 8：真正做登录判断的地方

文件：

- `backend/auth-service/src/main/java/com/finex/auth/service/impl/auth/AuthLoginDomainSupport.java`

它是干什么的：

- 真正执行登录校验：查用户、比密码、看状态、发 token。

谁会调用它：

- `UserServiceImpl`

它再调用谁：

- `UserMapper`
- `AuthAuthorizationDomainSupport`
- `JwtUtil`

输入 / 输出是什么：

- 输入：`LoginDTO`
- 输出：`LoginVO`

业务规则是什么：

- 按用户名查用户。
- 用 MD5 比对密码。
- 检查用户是否启用。
- 生成 JWT token。
- 补充角色和权限编码。

风险点是什么：

- 密码比对逻辑一旦出错，会导致无法登录或错误放行。
- 用户状态判断一旦缺失，被禁用用户可能还能登录。
- token 生成配置不对，会导致登录后很快失效或验证失败。

你只需要记住的一句话：

- “AuthLoginDomainSupport 才是真正判定能不能登录的人。”

---

## 文件卡片 9：数据库访问层

文件：

- `backend/auth-service/src/main/java/com/finex/auth/mapper/UserMapper.java`

它是干什么的：

- 负责读写用户表。

谁会调用它：

- 登录相关服务、权限相关服务

它再调用谁：

- 数据库

输入 / 输出是什么：

- 输入：查询条件
- 输出：`User`

业务规则是什么：

- 它不定义业务意义，只负责数据读写。

风险点是什么：

- 如果查错用户、查多条、漏过滤状态或字段，会影响登录和权限判断。

你只需要记住的一句话：

- “UserMapper 是后端去拿用户数据的数据库窗口。”

---

## 文件卡片 10：用户实体

文件：

- `backend/auth-service/src/main/java/com/finex/auth/entity/User.java`

它是干什么的：

- 描述 `sys_user` 表的一条用户记录在 Java 里的样子。

谁会调用它：

- `UserMapper`
- 各种 Service

它再调用谁：

- 它是数据对象，不主动调用别人。

输入 / 输出是什么：

- 主要字段包括：用户名、密码、姓名、手机号、邮箱、公司、部门、职位、状态等

业务规则是什么：

- `status` 对登录是否允许很重要。

风险点是什么：

- 如果关键字段名理解错，比如把 `status`、`companyId`、`deptId` 理解错，后面很多业务判断都会偏。

你只需要记住的一句话：

- “User 就是 sys_user 表在 Java 里的影子。”

---

## 文件卡片 11：登录返回结果

文件：

- `backend/auth-service/src/main/java/com/finex/auth/dto/LoginVO.java`
- `backend/common/src/main/java/com/finex/common/Result.java`

它们是干什么的：

- `LoginVO`：定义登录成功后返回哪些业务字段
- `Result`：定义所有接口统一回包格式

谁会调用它们：

- `AuthLoginDomainSupport`
- `AuthController`

输入 / 输出是什么：

- 登录成功后，前端会拿到 `userId`、`username`、`name`、`token`、`expireIn`、`roles`、`permissionCodes`

业务规则是什么：

- 页面不直接拿裸数据，而是拿统一格式的结果对象。

风险点是什么：

- 如果缺少角色或权限字段，后续菜单和权限控制会异常。
- 如果 `Result` 格式不一致，前端统一请求层会失效。

你只需要记住的一句话：

- “LoginVO 是登录成功后真正给前端看的内容，Result 是所有接口共用的外包装。”

---

## 文件卡片 12：JWT 工具类

文件：

- `backend/common/src/main/java/com/finex/common/JwtUtil.java`

它是干什么的：

- 生成和校验 token。

谁会调用它：

- 登录相关逻辑
- token 校验相关逻辑

它再调用谁：

- JWT 库和系统配置

输入 / 输出是什么：

- 输入：用户 ID、用户名
- 输出：token

业务规则是什么：

- token 包含用户身份信息。
- 依赖 `FINEX_JWT_SECRET` 或配置项中的密钥。

风险点是什么：

- JWT 密钥没配好，登录态会不稳定。
- 如果使用临时内存密钥，重启后旧 token 会失效。

你只需要记住的一句话：

- “JwtUtil 负责发登录通行证，也负责验证通行证。”

---

## 文件卡片 13：登录后的用户信息补充

文件：

- `frontend/admin-web/src/router/index.ts`
- `backend/auth-service/src/main/java/com/finex/auth/controller/MvpController.java`

它们是干什么的：

- 前端路由守卫在进入受保护页面时，检查本地有没有 token 和权限信息。
- 如果本地没有完整权限，会请求 `/auth/me`。
- `/auth/me` 实际由 `MvpController.me` 提供。

谁会调用它：

- 前端路由守卫

它再调用谁：

- `authApi.getCurrentUser`
- `mvpDataService.getCurrentUser`

输入 / 输出是什么：

- 输入：当前 token
- 输出：当前用户资料和权限

业务规则是什么：

- 登录成功不等于权限信息一定已经完整装载。
- 页面跳转前还会做一次补充加载和权限放行判断。

风险点是什么：

- 如果 `/auth/me` 出问题，用户可能“已登录但进不去页面”。
- 如果权限码不完整，菜单和路由放行会异常。

你只需要记住的一句话：

- “登录成功只是第一步，路由守卫还会继续补当前用户和权限。”

---

## 这条链路里最重要的 5 个业务判断点

1. 前端有没有把用户名和密码正确发给后端。
2. 后端有没有正确查到用户。
3. 密码比对和用户状态判断有没有正确执行。
4. token 有没有正确生成并保存。
5. 登录后当前用户和权限有没有被正确补齐。

---

## 财务视角摘要

从财务角度看，登录链路不是“技术小问题”，它直接影响：

- 谁能进入系统
- 谁能看到哪些菜单
- 谁能做哪些操作
- 权限是否会误放行

如果这条链路出错，常见业务后果是：

- 应该能登录的人登不进去
- 已禁用的人还在继续操作
- 登录后权限不完整，页面进不去
- 权限错配，看到不该看的菜单或单据

---

## 你以后审这条链路时优先看什么

如果以后你让我审登录相关改动，我会优先看：

- 参数校验有没有丢
- 密码比对是否被改坏
- 状态校验是否还在
- token 生成逻辑是否变化
- `/auth/me` 和权限补齐链路是否一致
- 前端是否正确处理 401 和 token 失效
