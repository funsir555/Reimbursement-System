# 第 1 回合：全项目导览

这一回合的目标不是看懂细节，而是建立“整条链路长什么样”的感觉。

你只需要回答一个问题：

“一个业务页面从前端到后端再到数据库，大概会经过哪些层？”

---

## 本回合要解决的业务问题

先不管具体是登录、报销还是审批。

你只需要先理解这件事：

“系统里任何一个功能，通常都不是一个文件单独完成的，而是多层协作完成的。”

---

## 全局链路图

先记住这一条总链路：

`浏览器页面 -> 路由 -> 页面文件 -> 前端 API -> 网关/后端接口 -> Service -> Mapper -> 数据库`

你可以把它翻译成财务语言：

`操作页面 -> 找到入口 -> 提交单据 -> 后台受理 -> 按规则处理 -> 查/改数据库 -> 返回结果`

## 按层理解整个项目

## 1. 前端启动层

文件：

- `frontend/admin-web/src/main.ts`

这一层负责把前端应用启动起来。

你可以把它理解成：

- 把整个网页后台点亮
- 把路由、状态管理、UI 框架注册好

它不是业务规则层，但它是前端系统的启动点。

## 2. 路由层

文件：

- `frontend/admin-web/src/router/index.ts`
- `frontend/admin-web/src/router/route-catalog.ts`

这一层负责：

- 页面地址怎么对应到页面文件
- 用户有没有 token
- 用户有没有权限访问某个页面

你可以把它理解成：

- 页面入口表
- 页面放行规则

## 3. 页面层

文件：

- `frontend/admin-web/src/views/...`

这一层负责：

- 展示输入框、按钮、表格
- 接收用户操作
- 在用户点击后触发 API 请求

它最接近业务用户。

## 4. 前端 API 层

文件：

- `frontend/admin-web/src/api/modules/*.ts`

这一层负责：

- 指定请求地址
- 指定请求方法
- 把参数打包给后端
- 带上 token

它是前端和后端的接口胶水层。

## 5. 后端 Controller 层

文件：

- `backend/auth-service/src/main/java/com/finex/auth/controller/*.java`

这一层负责：

- 接住前端发来的请求
- 收参数
- 调用 Service
- 把结果包装后返回给前端

它通常不是业务最深处，但它是后端入口。

## 6. 后端 Service 层

文件：

- `backend/auth-service/src/main/java/com/finex/auth/service/impl/*.java`

这一层负责：

- 校验业务规则
- 决定能不能做
- 调用数据库读写
- 组合多个步骤

这一层是你以后做业务审查时最值得重点看的层。

## 7. Mapper / Entity / DTO / VO 层

文件：

- `mapper`
- `entity`
- `dto`

这一层负责：

- 和数据库表打交道
- 约束前后端传递的数据格式
- 定义“数据长什么样”

## 8. 配置与数据库脚本层

文件：

- `backend/auth-service/src/main/resources/application.yml`
- `backend/sql/*.sql`

这一层负责：

- 决定系统连哪个数据库、用什么端口、用什么密钥
- 决定数据库表结构长什么样

---

## 你在这一回合最该形成的理解

以后你看到一个业务页面，不要立刻盯死页面代码。

你应该默认问自己：

1. 这个页面是通过哪个路由进入的？
2. 它调用了哪个前端 API？
3. 这个 API 调到哪个后端 Controller？
4. 哪个 Service 在真正处理业务？
5. 它最后可能改哪张表？
6. 有没有测试在保护这个规则？

只要这个问题顺序建立起来，你后面看任何模块都会容易很多。

---

## 这回合先不深入的内容

先不要深挖这些：

- 某个复杂页面的所有交互细节
- 某个 Service 里所有分支
- 每一张数据库表的所有字段
- 每一个测试细节

你现在的任务只是先建立“层次感”，不是建立“细节全量记忆”。

---

## 财务视角摘要

从财务角度看，这一回合真正要学的是：

- 什么叫“前台展示”和“后台处理”分开。
- 什么叫“业务规则不一定写在页面里，而通常写在 Service 里”。
- 什么叫“数据库不是随便改的，它受 Mapper、Entity 和 SQL 一起约束”。
- 什么叫“权限与登录不只是页面行为，还受后端控制”。

这会直接影响你以后做审查时的判断：

- 页面上看起来能点，不代表后台一定允许。
- 后台接口能调用，不代表数据库就一定会正确落库。
- 看到一个 bug，也不能只看页面，必须沿调用链追到后端。

## 你下一回合会进入什么

下一回合开始，我们不再停留在全局层次，而是进入一条最短、最适合入门的真实链路：

`LoginView.vue -> auth.ts -> AuthController -> UserServiceImpl -> AuthLoginDomainSupport -> UserMapper/User -> JwtUtil -> Result`

也会顺带补一个很重要的事实：

登录成功后，前端后面还会通过 `/auth/me` 去补当前用户信息，而这个接口实际在 `MvpController`。

这能帮助你建立“一个前端模块，不一定只对应一个后端 Controller”的理解。
