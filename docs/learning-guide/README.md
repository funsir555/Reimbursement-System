# 非开发者学习文档索引

这套文档是给“不会 Java、主要从财务和业务视角理解系统”的你准备的。

目标不是让你先学会写代码，而是先学会三件事：

1. 看懂一个文件大概负责什么。
2. 看懂一条业务链的数据是怎么流动的。
3. 在我的帮助下，开始做业务风险导向的代码审查。

---

## 这套文档怎么用

请按顺序阅读，不要一上来试图把整个项目所有文件都看完。

推荐顺序：

- 只有当你能用自己的话复述“这段链路是谁调用谁、影响什么业务”时，再进入下一份。

## 你会先学到什么

第一阶段不是 Java 语法课，而是“业务流程 + 文件职责”课。

你会先理解：

- 前端从哪里进入：页面、路由、接口。
- 后端从哪里进入：Controller、Service、Mapper。
- 数据最后落在哪里：Entity、数据库表、SQL。
- 测试文件为什么存在：它们是在保护什么业务规则。

这对应你后续最常见的两个问题：

- “这个文件是干什么的？”
- “如果改坏了，会影响什么单据、权限、金额或状态？”

## 学习时的固定阅读顺序

以后无论你看哪个模块，默认都按这条顺序走：

`路由/router -> 页面/views -> 前端接口/api -> 后端Controller -> Service -> Mapper/Entity/DTO/VO -> SQL/表 -> 测试`

你不需要一次记住所有细节，只要记住一句话：

“先找入口，再看中转，再看真正做事的地方，最后看数据落点和保护措施。”

## 收口填写怎么用

每份文档最后都有一块统一的“收口填写区”。

你学完后请手动填写，不用管格式好不好，重点是把自己的理解写出来。

你可以把它当成你的学习验收单。只要能写出来，通常就说明你真的开始理解了。

推荐填写标准：

- “本回合学会了什么”：写你已经能复述的内容。
- “仍然不懂什么”：写最卡住你的 1-3 个点。
- “我能否复述调用链”：写“能/不能”，如果能就写出一句完整链路。
- “我认为最大的业务风险点”：从财务视角写风险，而不是从代码风格写。
- “是否进入下一回合”：只有在你能复述时再写“是”。

## 你之后怎么和我配合

你可以直接把以下内容丢给我：

- 一个文件路径
- 一个模块名
- 一段报错
- 一次改动的 diff
- 一页界面
- 你自己写的收口内容

我会按固定方式帮你：

1. 先翻译成人话。
2. 再讲上下游调用关系。
3. 再讲它影响什么业务。
4. 最后提醒你风险点和审查点。
