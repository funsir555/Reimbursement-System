# 代码审查入门

这份文档的目标不是把你变成“挑代码风格的人”，而是把你变成“能抓业务风险的人”。

对你来说，代码审查最重要的不是：

- 缩进漂不漂亮
- 命名高级不高级

而是：

- 权限有没有放错
- 金额和状态会不会错
- 单据会不会重复
- 数据会不会被弄脏

---

## 先记住一句话

你的审查顺序应该永远是：

`业务正确性 > 数据一致性 > 权限安全 > 可追溯性 > 代码风格`

---

## 你以后可以怎么把材料发给我

你可以直接把下面任何一种东西发给我：

- 一个文件路径
- 一个模块名
- 一次页面功能改动
- 一个报错信息
- 一个 Git diff
- 一段 SQL
- 你自己写的“收口填写区”内容

我会默认按“发现优先”格式回答你。

---

## 我给你的审查输出格式

以后你让我审查时，我会尽量按这个顺序给你：

1. 问题等级
2. 文件位置
3. 大概是哪段逻辑
4. 为什么这是风险
5. 会影响什么业务
6. 你怎么验证

如果我没发现明确问题，我也会直说，并提醒剩余风险和测试空白。

## 你的财务审查清单

你先不要从“代码写得好不好”看，而是先问下面这些问题。

### 1. 权限对不对

你要问：

- 这个操作是不是谁都能做？
- 被禁用或不该看的人，会不会还能访问？
- 菜单有权限，后端接口有没有同步做权限限制？

### 2. 金额和税额对不对

你要问：

- 金额字段有没有漏传、漏算、重复算？
- 税额、价税合计、核销金额会不会不一致？
- 金额字段有没有可能被空值、字符串、精度问题影响？

### 3. 单据状态会不会错

你要问：

- 草稿、提交、审批中、已驳回、已付款、已生成凭证，这些状态有没有乱跳？
- 状态变化是不是只能按制度规定的方向走？
- 某一步失败时，会不会留下半成品状态？

### 4. 会不会重复

你要问：

- 会不会重复提交单据？
- 会不会重复审批？
- 会不会重复付款？
- 会不会重复生成凭证？
- 会不会重复生成附件或下载记录？

### 5. 留痕够不够

你要问：

- 关键动作有没有日志、记录、附件、下载记录、操作轨迹？
- 改了系统设置后，旧单据还能不能追溯当时的规则？

## 我的技术审查清单

当你把材料发给我时，我通常会重点看这些：

### 1. 参数校验

- DTO 是否校验非空、格式、范围
- 前端与后端对字段理解是否一致

### 2. 空值和异常处理

- 查不到数据怎么办
- 数据库失败怎么办
- 外部接口失败怎么办
- 是否会把异常直接暴露给前端

### 3. 事务一致性

- 一次业务动作如果改了多张表，会不会只改成功一半
- 失败时有没有回滚

### 4. 权限控制

- 只是前端限制，还是后端也限制了
- Controller 和 Service 是否都做了必要保护

### 5. 并发与重复提交

- 快速点击两次按钮会怎样
- 重试会不会重复创建记录

### 6. SQL 条件范围

- 查询会不会查多
- 更新会不会改多
- 条件是否可能漏带公司、用户、状态等关键约束

### 7. 日志与可追溯性

- 关键动作有没有留下足够线索
- 后续排错时能不能知道是谁、何时、改了什么

### 8. 测试覆盖

- 成功分支有没有测
- 失败分支有没有测
- 权限分支有没有测
- 边界条件有没有测

## 你最适合问我的两类问题

当你还不会判断代码时，最实用的是直接问我：

### 问法 1

“这段代码业务上在防什么风险？”

适合你想知道：

- 为什么这里要校验
- 为什么这里要判断状态
- 为什么这里要加权限

### 问法 2

“如果删掉或改掉这里，最可能坏在哪里？”

适合你想知道：

- 这段逻辑是不是关键控制点
- 会影响哪些页面、单据、状态或数据

---

## 你以后看改动时的固定步骤

你可以按这个顺序检查任何改动：

1. 这次改动入口在哪个页面或接口？
2. 它改的是显示，还是改的是业务规则？
3. 它最终会影响哪张表、哪类单据、哪种状态？
4. 有没有权限判断？
5. 有没有重复执行风险？
6. 失败时会不会留下脏数据？
7. 有没有测试或至少有手工验证路径？

```text
2. 金额/税额
3. 状态流转
4. 重复提交/重复生成
5. 数据一致性
6. 是否有必要测试
```

## 什么时候你可以说“我开始会审了”

当你能独立提出下面任意 3 个问题时，你就已经开始具备审查能力了：

- 这个改动会不会让不该看的人看到数据？
- 这个改动会不会让单据状态跳错？
- 这个改动会不会导致重复提交或重复生成凭证？
- 这个改动失败时会不会留下半成品数据？
- 这个改动会不会影响旧单据追溯？

你不需要先会写 Java，先会问对问题，就已经非常有价值。
